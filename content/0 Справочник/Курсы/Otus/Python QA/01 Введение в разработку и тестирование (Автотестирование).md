## ⚡ Что такое автотестирование

> [!important] Определение **Автотестирование** — это **часть процесса разработки**.

Автотестирование — это не просто «записать действия и воспроизвести». Это написание кода, который:

- проверяет работу приложения автоматически,
- может запускаться многократно без участия человека,
- интегрируется в процессы CI/CD.

> [!quote] Об автоматизации и мышлении разработчика «Автотестирование — не для того, чтобы вы, как ручные тестировщики, которым показали, как на BDD тесты писать с каким-нибудь RSpec. А в том, чтобы вы думали, в том числе как разработчики.»

> [!note] Уточнение «Как разработчики» — не значит «обезьянки». Смысл в том, чтобы **понимать внутренние механизмы**, а не просто механически писать тесты.

## 🛠 Архитектура тестового фреймворка

> [!important] Суть архитектуры **Архитектура** — это то, **как хранятся файлы в вашем проекте**. Никакой магии — просто правильная организация структуры папок и файлов.

### 📋 Структура реального проекта (пример лектора)

Лектор показал структуру реального тестового проекта, который поддерживает периодически. Проект написан на **Python 3.7**, используется **PyCharm**.

|Папка / Файл|Назначение|
|---|---|
|`allurereport/`|Подключён Allure Report — отчётность, история, плагины|
|`client/` (или `server/`)|Взаимодействие с запросами, взаимодействие с базой данных|
|`core/`|Основные функции: драйверы (ChromeDriver, GeckoDriver), основные шаги, описания функций|
|`reports/`|Отдельная папка с отчётами|
|`scenarios/`|Тестовые сценарии (порядка 200+ штук в реальном проекте)|
|`utils/`|Утилиты — небольшие вспомогательные функции, которые постоянно нужны в тестах|
|`locators/`|Локаторы, вынесенные в отдельную папку (хорошая практика)|
|`logger/`|Логгер, который пишет логи в базу данных|
|`.gitignore`|Список файлов, изменения в которых Git не отслеживает|
|`browsers.json`|Настройки браузеров|
|`conftest.py`|Основной файл для запуска (конфигурация pytest)|
|`pytest.ini`|Настройки PyTest, с которыми он запускается|
|`requirements.txt`|Список библиотек для установки (`pip install -r requirements.txt`)|
|`README.md`|Описание проекта|
|`test.bat`|Батник для запуска (решение разработчиков для портируемости на CI-системы)|

> [!tip] Про `requirements.txt` Файл `requirements.txt` содержит все используемые библиотеки. Позволяет быстро развернуть окружение на новом сервере:
> 
> ```bash
> pip install -r requirements.txt
> ```

> [!note] Про `.gitignore` `.gitignore` нужен для того, чтобы **Git не видел изменения** в определённых файлах (например, локальные настройки, кэш, секреты).

> [!tip] Про локаторы Хранить локаторы в отдельной папке — **хорошая практика**. Используется не всегда, зависит от проекта.

> [!quote] Лектор про `utils` «Утилиты — это какие-то небольшие функции, которые постоянно пригождаются в тестах. Например, у меня в `tools` была штука на JavaScript, которая скроллила страницы куда мне надо по определённым критериям.»

## 💡 Понимание внутренних механизмов — ключ к росту

> [!important] Что отличает специалистов разного уровня Понимание **внутренних механизмов** — как протоколы работают на каком уровне, как взаимодействовать с этими протоколами, как всё устроено внутри — **отличает джуниора от мидла, мидла от сеньора, сеньора от лида**.

- Нужно **evidence-понимание** того, на какие вопросы вы отвечаете на собеседованиях.
- Важно понимать, какие **протоколы** на каком уровне работают.
- Ключевое умение реального middle/middle+ в автоматизации — **troubleshooting**: разбираться, где что сломалось и почему.

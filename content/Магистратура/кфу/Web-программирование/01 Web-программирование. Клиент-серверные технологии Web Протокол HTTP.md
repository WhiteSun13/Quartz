### **1. Введение: Как работает Web?**

Каждый раз, когда вы вводите адрес сайта в браузере и нажимаете Enter, вы запускаете сложный, но строго регламентированный процесс обмена данными. В основе этого процесса лежит **клиент-серверная архитектура**, а языком общения служит протокол **HTTP**. Понимание этих двух столпов необходимо любому, кто связан с созданием или поддержкой веб-приложений.

*   **Всемирная паутина (World Wide Web)** – это не сам Интернет. Интернет – это глобальная сеть физически соединенных компьютеров. Web – это информационная система, работающая поверх Интернета, которая позволяет получать доступ к связанным друг с другом документам (веб-страницам) с помощью браузеров.
*   **Основные компоненты:**
    *   **Клиент:** Программа, которая запрашивает информацию (например, ваш браузер).
    *   **Сервер:** Мощный компьютер, который хранит информацию и предоставляет её по запросу.
    *   **Протокол:** Набор правил, по которым клиент и сервер общаются друг с другом. Главный протокол в Web — это **HTTP**.

### **2. Глава 1: Клиент-серверная архитектура**

#### 2.1. Определение клиент-серверной модели

**Клиент-серверная модель** – это сетевая архитектура, в которой задачи или рабочая нагрузка распределены между поставщиками услуг, называемыми **серверами**, и заказчиками услуг, называемыми **клиентами**.

Аналогия: Ресторан.
*   **Вы (клиент)** приходите в ресторан и делаете заказ (запрос).
*   **Официант (сеть/протокол)** передает ваш заказ на кухню.
*   **Кухня (сервер)** готовит блюдо (обрабатывает запрос) и отдает его официанту.
*   **Официант** приносит вам готовое блюдо (ответ).

#### 2.2. Роль и задачи клиента

**Клиент** – это инициатор взаимодействия. Его главная задача – формировать и отправлять запросы на сервер для получения данных или выполнения каких-либо действий.

*   **Примеры клиентов:**
    *   Веб-браузеры (Chrome, Firefox, Safari).
    *   Мобильные приложения.
    *   Программы для работы с API (Postman, Insomnia).
    *   Консольные утилиты (`curl`, `wget`).
    *   Поисковые роботы (Googlebot, YandexBot).
*   **Основные функции:**
    1.  Формирование HTTP-запроса на основе действий пользователя (клик по ссылке, отправка формы).
    2.  Отправка этого запроса на сервер.
    3.  Получение HTTP-ответа от сервера.
    4.  Интерпретация (парсинг) и отображение (рендеринг) полученных данных. Например, HTML-код превращается в видимую веб-страницу, JSON-данные используются для обновления интерфейса.

#### 2.3. Роль и задачи сервера

**Сервер** – это пассивный участник, который постоянно ожидает запросов от клиентов.

*   **Компоненты сервера:**
    *   **Аппаратное обеспечение:** Физический компьютер с процессором, памятью и диском.
    *   **Операционная система:** (например, Linux, Windows Server).
    *   **Веб-сервер (ПО):** Программа, которая принимает HTTP-запросы и отдает на них ответы. Самые популярные: **Nginx, Apache**.
    *   **Прикладная логика:** Код (написанный на PHP, Python, Java, Node.js и т.д.), который обрабатывает бизнес-логику запроса (например, поиск товара в базе данных, авторизация пользователя).
    *   **База данных:** Система для хранения данных (MySQL, PostgreSQL, MongoDB).
*   **Основные функции:**
    1.  Прослушивание сетевого порта (обычно 80 для HTTP и 443 для HTTPS) в ожидании входящих соединений.
    2.  Принятие и парсинг HTTP-запроса от клиента.
    3.  Обработка запроса: выполнение кода, обращение к базе данных, чтение файлов.
    4.  Формирование HTTP-ответа, включающего статус, заголовки и тело.
    5.  Отправка ответа клиенту.

#### 2.4. Пошаговая схема взаимодействия

1.  **Пользователь** вводит в браузере `https://www.example.com/products` и нажимает Enter.
2.  **Браузер (клиент)** определяет доменное имя (`www.example.com`).
3.  **DNS-запрос:** Браузер обращается к системе доменных имен (DNS), чтобы узнать IP-адрес, соответствующий `www.example.com`. Допустим, DNS отвечает `93.184.216.34`.
4.  **Установка соединения:** Браузер устанавливает с сервером по IP-адресу `93.184.216.34` соединение по протоколу TCP на порт 443 (так как используется HTTPS).
5.  **Отправка HTTP-запроса:** Браузер формирует и отправляет HTTP-запрос, который выглядит примерно так:
    
```http
    GET /products HTTP/1.1
    Host: www.example.com
    User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64) ...
    Accept: text/html,...
```
    
6.  **Обработка на сервере:** Веб-сервер (Nginx) на `93.184.216.34` получает запрос. Он видит, что запрашивается путь `/products`, и передает запрос на обработку соответствующему приложению (например, скрипту на Python).
7.  **Приложение** обращается к базе данных, получает список продуктов, генерирует из них HTML-страницу.
8.  **Формирование HTTP-ответа:** Приложение возвращает сгенерированную HTML-страницу веб-серверу, который формирует полный HTTP-ответ:

```http
    HTTP/1.1 200 OK
    Content-Type: text/html; charset=UTF-8
    Content-Length: 15432
    Date: Wed, 21 Oct 2023 07:28:00 GMT

    <!DOCTYPE html><html><head>...</head><body>...</body></html>
```

9.  **Отправка ответа:** Сервер отправляет этот ответ обратно клиенту.
10. **Рендеринг:** Браузер получает ответ. Код состояния `200 OK` говорит, что всё прошло успешно. Браузер начинает рендерить (отображать) полученный HTML-код. Если в HTML есть ссылки на другие ресурсы (CSS, JS, картинки), браузер выполнит шаги 4-9 для каждого из них.

### **3. Глава 2: Протокол HTTP – Язык общения в Web**

#### 3.1. Что такое протокол?

**Протокол** — это набор правил и соглашений, которые определяют, как два или более устройства в сети должны обмениваться данными. Он регламентирует синтаксис, семантику и синхронизацию коммуникации. Без протоколов общение в сети было бы хаотичным и невозможным.

#### 3.2. HTTP: определение и ключевые свойства

**HTTP (HyperText Transfer Protocol — протокол передачи гипертекста)** — это прикладной протокол для передачи данных, изначально в виде гипертекстовых документов в формате HTML. Сегодня по HTTP передаются любые данные: HTML, CSS, JavaScript, изображения, видео, JSON, XML и т.д.

**Ключевые свойства:**

1.  **Текстовый формат:** Сообщения HTTP (до версии HTTP/2) представляют собой обычный текст, который легко читать и отлаживать человеку.
2.  **Модель "Запрос-Ответ" (Request-Response):** Коммуникация всегда инициируется клиентом. Клиент отправляет запрос, сервер отвечает. Сервер никогда не может инициировать связь с клиентом самостоятельно.
3.  **Отсутствие состояния (Stateless):** Это **фундаментальное** свойство. Каждый HTTP-запрос рассматривается сервером как полностью независимый. Сервер не помнит ничего о предыдущих запросах от того же клиента.
    *   **Проблема:** Как реализовать корзину в интернет-магазине или авторизацию, если сервер "не помнит" пользователя?
    *   **Решение:** Для поддержания состояния используются дополнительные механизмы, такие как **Cookies**, **сессии** на сервере или **токены** (JWT). Клиент (браузер) сохраняет cookie и прикрепляет их ко всем последующим запросам к этому же серверу, позволяя серверу идентифицировать пользователя.

#### 3.3. Структура HTTP-сообщения

Существует два типа сообщений: **запрос** (от клиента к серверу) и **ответ** (от сервера к клиенту). Оба имеют схожую структуру:

1.  **Стартовая строка** (Start Line / Status Line)
2.  **Заголовки** (Headers)
3.  **Пустая строка** (отделяет заголовки от тела)
4.  **Тело сообщения** (Body) - необязательное

### **4. Глава 3: Детальный разбор HTTP-запроса (HTTP Request)**

```http
(1) GET /search?q=http HTTP/1.1
(2) Host: www.google.com
    User-Agent: curl/7.64.1
    Accept: */*
(3)
(4)
```

#### 4.1. Стартовая строка (Start Line)

Состоит из трех частей: `Метод URI Версия`.

*   **Методы HTTP (HTTP Methods/Verbs):** Указывают на действие, которое клиент хочет выполнить с ресурсом.
    *   **GET:** Запрос представления ресурса. Самый частый метод. Используется для получения данных. Безопасен и идемпотентен (многократное повторение дает тот же результат).
    *   **POST:** Отправка данных на сервер для создания нового ресурса (например, отправка формы регистрации, создание поста в блоге). Не является идемпотентным.
    *   **PUT:** Полное обновление существующего ресурса. Если ресурса нет, может его создать. Идемпотентен.
    *   **DELETE:** Удаление указанного ресурса. Идемпотентен.
    *   **PATCH:** Частичное обновление ресурса. Не является идемпотентным.
    *   **HEAD:** Аналогичен GET, но в ответе сервера отсутствует тело. Используется для получения метаинформации (заголовков) о ресурсе, не скачивая его целиком.
    *   **OPTIONS:** Запрашивает информацию о доступных опциях коммуникации для целевого ресурса (например, какие методы разрешены). Используется в механизме CORS.

*   **URI (Uniform Resource Identifier):** Идентификатор ресурса. Чаще всего это путь к ресурсу на сервере (`/path/to/resource.html`) и параметры запроса (`?key1=value1&key2=value2`).

*   **Версия протокола:** `HTTP/1.1`, `HTTP/2`.

#### 4.2. Заголовки (Headers)

Пары `Ключ: Значение`, содержащие метаинформацию о запросе.

*   `Host`: **Обязательный** заголовок в HTTP/1.1. Указывает доменное имя сервера. Это позволяет одному веб-серверу обслуживать несколько сайтов на одном IP-адресе.
*   `User-Agent`: Информация о клиенте (браузере, ОС), который отправляет запрос.
*   `Accept`: Сообщает серверу, какие типы контента (MIME-типы) клиент может понять (например, `text/html`, `application/json`, `image/jpeg`).
*   `Accept-Language`: Предпочитаемый язык для ответа (`en-US`, `ru-RU`).
*   `Authorization`: Содержит данные для аутентификации клиента на сервере (например, `Bearer <token>`).
*   `Content-Type`: MIME-тип тела запроса. Важен для методов POST и PUT (например, `application/json` или `application/x-www-form-urlencoded`).
*   `Content-Length`: Размер тела запроса в байтах.
*   `Cookie`: Передает на сервер cookie, ранее установленные этим сервером.

#### 4.3. Тело запроса (Body)

Необязательная часть. Содержит данные, отправляемые на сервер. Используется в методах POST, PUT, PATCH. Тело может содержать что угодно: JSON-строку, XML, данные формы, бинарный код файла.

### **5. Глава 4: Детальный разбор HTTP-ответа (HTTP Response)**

```http
(1) HTTP/1.1 200 OK
(2) Date: Wed, 21 Oct 2023 07:28:00 GMT
    Server: gws
    Content-Type: text/html; charset=UTF-8
    Content-Length: 12345
    Set-Cookie: ID=...; expires=...;
(3)
(4) <!DOCTYPE html>
    <html>
      ...
    </html>
```

#### 5.1. Строка состояния (Status Line)

`Версия КодСостояния ТекстовоеОписание`

*   **Коды состояния (Status Codes):** Трехзначное число, информирующее клиента о результате обработки запроса. Они сгруппированы в 5 классов:
    *   **1xx (Informational):** Запрос принят, продолжается обработка. Редко используются.
    *   **2xx (Success):** Запрос был успешно принят и обработан.
        *   `200 OK`: Стандартный успешный ответ.
        *   `201 Created`: Запрос выполнен, и в результате был создан новый ресурс (например, после POST-запроса).
        *   `204 No Content`: Сервер успешно обработал запрос, но в ответе нет тела (например, после успешного DELETE-запроса).
    *   **3xx (Redirection):** Для выполнения запроса требуются дальнейшие действия со стороны клиента.
        *   `301 Moved Permanently`: Ресурс навсегда перемещен на новый URI. Браузеры кешируют этот редирект.
        *   `302 Found`: Временный редирект. Клиент должен использовать новый URI для этого запроса, но в будущем продолжать использовать старый.
        *   `304 Not Modified`: Используется для кеширования. Сообщает клиенту, что ресурс не изменился с момента последнего запроса, и можно использовать локальную копию.
    *   **4xx (Client Error):** Ошибка на стороне клиента.
        *   `400 Bad Request`: Сервер не может понять запрос из-за неверного синтаксиса.
        *   `401 Unauthorized`: Для доступа к ресурсу требуется аутентификация.
        *   `403 Forbidden`: Сервер понял запрос, но отказывается его выполнять. У клиента нет прав доступа (в отличие от 401, здесь аутентификация не поможет).
        *   `404 Not Found`: Самый известный код. Сервер не может найти запрошенный ресурс.
    *   **5xx (Server Error):** Ошибка на стороне сервера.
        *   `500 Internal Server Error`: Внутренняя ошибка сервера. Произошло что-то непредвиденное, что помешало выполнить запрос.
        *   `503 Service Unavailable`: Сервер временно не может обрабатывать запросы (перегружен или на техобслуживании).

#### 5.2. Заголовки (Headers)

*   `Content-Type`: MIME-тип данных в теле ответа (например, `text/html`, application/json).
*   `Content-Length`: Размер тела ответа в байтах.
*   `Server`: Информация о веб-сервере, который сгенерировал ответ.
*   `Set-Cookie`: Устанавливает cookie в браузере клиента.
*   `Cache-Control`: Инструкции по кешированию ответа (`no-cache`, `max-age=3600`).
*   `Location`: Используется в ответах с кодом 3xx для указания нового URI.

#### 5.3. Тело ответа (Body)

Содержит запрошенный ресурс: HTML-страница, JSON-объект, изображение и т.д.

### **6. Глава 5: Эволюция и безопасность**

#### 6.1. Отличия версий HTTP

*   **HTTP/1.0 (1996):** Простой. Устанавливал новое TCP-соединение для каждого запроса/ответа, что было очень неэффективно.
*   **HTTP/1.1 (1999):** Стандарт на долгие годы. Ввёл:
    *   **Постоянные соединения (Persistent Connections):** Позволяет отправлять несколько запросов/ответов в рамках одного TCP-соединения.
    *   **Конвейеризация (Pipelining):** Позволяла отправлять несколько запросов, не дожидаясь ответа на предыдущие (но ответы должны были приходить строго по порядку).
*   **HTTP/2 (2015):** Революционное обновление для повышения производительности.
    *   **Бинарный протокол:** Вместо текста используются бинарные фреймы.
    *   **Мультиплексирование:** Позволяет асинхронно отправлять множество запросов и ответов в одном TCP-соединении без блокировки "началом очереди" (Head-of-line blocking).
    *   **Сжатие заголовков (HPACK):** Эффективно сжимает часто повторяющиеся заголовки.
    *   **Server Push:** Сервер может отправлять ресурсы клиенту по своей инициативе, до того как клиент их запросит.
*   **HTTP/3 (2022):** Новейший стандарт, работающий поверх протокола **QUIC** вместо TCP. Решает проблему Head-of-line blocking на транспортном уровне, обеспечивает более быстрое установление соединения.

#### 6.2. HTTPS: Шифрование и безопасность

**HTTPS (HyperText Transfer Protocol Secure)** — это не отдельный протокол, а расширение HTTP, которое использует криптографические протоколы **SSL/TLS** для защиты соединения.

*   **SSL/TLS (Secure Sockets Layer / Transport Layer Security):** Протоколы, обеспечивающие три основных аспекта безопасности:
    1.  **Шифрование:** Все данные между клиентом и сервером шифруются, что делает их нечитаемыми для третьих лиц.
    2.  **Аутентификация:** Клиент может быть уверен, что общается именно с тем сервером, за который тот себя выдает, благодаря SSL-сертификату.
    3.  **Целостность данных:** Гарантирует, что данные не были изменены во время передачи.

*   **Как работает HTTPS (упрощенно):**
    1.  Клиент подключается к серверу на порт 443.
    2.  Происходит "рукопожатие" (TLS Handshake).
    3.  Сервер отправляет клиенту свой публичный SSL-сертификат.
    4.  Клиент проверяет подлинность сертификата у доверенного центра сертификации.
    5.  Клиент и сервер договариваются об общем секретном ключе для шифрования сессии.
    6.  Весь последующий HTTP-трафик шифруется этим ключом.

**HTTPS сегодня является обязательным стандартом** для всех сайтов, поскольку он защищает данные пользователей и повышает доверие.

### **7. Глава 6: Практическая работа**

#### 7.1. Анализ трафика в браузере

1.  Откройте любой сайт (например, `yandex.ru`).
2.  Нажмите `F12` (или `Ctrl+Shift+I`) для открытия инструментов разработчика.
3.  Перейдите на вкладку **"Сеть" (Network)**.
4.  Обновите страницу (`F5`).
5.  Вы увидите список всех HTTP-запросов, которые сделал браузер.
6.  Кликните на любой запрос (например, на самый первый, с именем домена).
7.  В правой панели вы увидите всю информацию:
    *   **General:** URL, метод, код состояния.
    *   **Response Headers:** Заголовки ответа от сервера.
    *   **Request Headers:** Заголовки, которые отправил ваш браузер.
    *   **Response/Preview:** Тело ответа.

#### 7.2. Использование утилиты `curl`

`curl` — это мощная консольная утилита для отправки HTTP-запросов.

*   **Простой GET-запрос:**
    ```bash
    curl https://api.github.com/users/octocat
    ```
*   **Просмотр заголовков ответа (`-i`):**
    ```bash
    curl -i https://www.google.com
    ```
*   **Полная трассировка, включая заголовки запроса (`-v`):**
    ```bash
    curl -v https://www.google.com
    ```
*   **POST-запрос с JSON-телом:**
    ```bash
    curl -X POST \
      -H "Content-Type: application/json" \
      -d '{"name": "New User", "job": "Developer"}' \
      https://reqres.in/api/users
    ```

#### 7.3. Пример простого web-сервера (концептуальный код)

**Python (с использованием фреймворка Flask):**
```python
# pip install Flask
from flask import Flask, jsonify

app = Flask(__name__)

# Обработчик для GET-запроса по адресу /
@app.route("/")
def hello_world():
    # Возвращает ответ с телом "Hello, World!" и кодом 200 OK
    return "<p>Hello, World!</p>"

# Обработчик для GET-запроса по адресу /api/data
@app.route("/api/data")
def get_data():
    user_data = {
        "id": 123,
        "username": "web_student",
        "role": "learner"
    }
    # Flask автоматически установит Content-Type: application/json
    return jsonify(user_data)

if __name__ == '__main__':
    app.run(debug=True, port=5000)
```
Этот код запускает локальный сервер. При обращении в браузере на `http://127.0.0.1:5000/api/data` вы увидите JSON-ответ, сформированный сервером.

### **9. Глоссарий**

*   **API (Application Programming Interface):** Интерфейс, который позволяет одной программе взаимодействовать с другой. Web API часто работают по HTTP.
*   **DNS (Domain Name System):** Система, преобразующая доменные имена в IP-адреса.
*   **JSON (JavaScript Object Notation):** Легковесный текстовый формат для обмена данными.
*   **MIME-тип:** Стандарт, определяющий тип и формат документа/файла (например, `text/html`, `image/png`).
*   **TCP (Transmission Control Protocol):** Протокол транспортного уровня, обеспечивающий надежную доставку данных.
*   **URI/URL (Uniform Resource Identifier/Locator):** Строка, однозначно идентифицирующая ресурс в сети.

Отлично, переходим к самому сердцу реляционных баз данных. Это фундаментальная тема, и ее понимание — ключ к работе с большинством современных СУБД.

### **Реляционная модель данных: введение**

**Реляционная модель данных** — это логическая модель данных, предложенная в 1970 году британским ученым Эдгаром Ф. Коддом. На сегодняшний день это самая распространенная модель в мире.

**Основная идея:** все данные в базе данных представляются в виде набора **отношений**. В практическом смысле, для пользователя **отношение** — это просто **таблица**.

Модель основана на строгом математическом аппарате — теории множеств и реляционной алгебре, что обеспечивает ей предсказуемость и надежность.

### **Основные понятия (математическая основа)**

Давайте разберем формальные термины и их практические аналоги.

| Формальный термин | Практический аналог | Описание |
| :----------------- | :------------------ | :----------------------------------------------------------------------------------------------------------------------------------------- |
| **Отношение**      | Таблица             | Набор данных об однотипных сущностях, состоящий из заголовка и тела.                                                                       |
| **Кортеж**         | Строка (запись)     | Один конкретный экземпляр сущности, представленный в виде набора значений его атрибутов.                                                      |
| **Атрибут**        | Столбец (поле)      | Именованная характеристика (свойство) сущности.                                                                                            |
| **Домен**          | Тип данных + правила | Множество всех допустимых значений, которые может принимать атрибут. Определяет "правила игры" для столбца.                               |

#### **1. Домен (Domain)**

Домен — это "поставщик" значений для атрибутов. Он определяет не только тип данных (например, целое число, строка), но и возможный диапазон или набор значений.

*   **Пример 1:** Для атрибута `ВозрастСотрудника` доменом может быть "целые числа от 18 до 65".
*   **Пример 2:** Для атрибута `Пол` доменом может быть "множество из двух значений: {'М', 'Ж'}".
*   **Пример 3:** Для атрибута `Email` доменом будет "строки, соответствующие формату электронной почты".

**Важно:** Все значения в одном столбце (атрибуте) должны принадлежать одному и тому же домену.

#### **2. Атрибут (Attribute)**

Атрибут — это именованный столбец отношения. Он описывает одно из свойств сущности, о которой мы храним информацию. Заголовок отношения состоит из набора атрибутов.

*   **Пример:** В отношении `Студенты` атрибутами могут быть: `ID_Студента`, `Имя`, `Фамилия`, `ДатаРождения`.

#### **3. Кортеж (Tuple)**

Кортеж — это одна строка в таблице. Он представляет собой упорядоченный набор значений атрибутов для одного конкретного объекта.

*   **Пример:** `(101, 'Иван', 'Иванов', '2002-05-15')` — это один кортеж в отношении `Студенты`.

#### **4. Отношение (Relation)**

Отношение — это целиком таблица, состоящая из заголовка (набора атрибутов) и тела (множества кортежей).

**Ключевые свойства отношения:**
1.  **Нет одинаковых кортежей:** В таблице не может быть двух абсолютно идентичных строк. Каждая строка уникальна.
2.  **Атомарность значений:** Каждая ячейка таблицы (пересечение строки и столбца) содержит только одно, неделимое значение.
3.  **Порядок кортежей не важен:** Порядок строк в таблице не имеет значения. Результат запроса не должен зависеть от того, в каком порядке хранятся строки.
4.  **Порядок атрибутов не важен (теоретически):** Столбцы можно переставлять местами без потери информации. На практике в SQL порядок важен для некоторых операций (например, `INSERT` без указания столбцов).

### **Типы данных в БД**

Домены реализуются в СУБД с помощью конкретных типов данных. Вот самые распространенные категории:

1.  **Числовые:**
    *   `INT` (или `INTEGER`): Целые числа.
    *   `BIGINT`: Целые числа большого диапазона.
    *   `DECIMAL` (или `NUMERIC`): Числа с фиксированной точностью, идеальны для финансовых расчетов.
    *   `FLOAT`, `REAL`: Числа с плавающей точкой.

2.  **Строковые:**
    *   `CHAR(n)`: Строка фиксированной длины `n`. Если строка короче, она дополняется пробелами.
    *   `VARCHAR(n)`: Строка переменной длины до `n` символов. Занимает столько места, сколько символов в ней есть.
    *   `TEXT`: Строка большой длины без фиксированного ограничения.

3.  **Дата и время:**
    *   `DATE`: Хранит дату (год, месяц, день).
    *   `TIME`: Хранит время (часы, минуты, секунды).
    *   `TIMESTAMP`: Хранит и дату, и время.

4.  **Логические:**
    *   `BOOLEAN`: Может принимать значения `TRUE`, `FALSE` или `NULL`.

5.  **Бинарные:**
    *   `BLOB` (Binary Large Object): Для хранения двоичных данных, таких как изображения, аудиофайлы.

### **Целостность реляционных данных**

Целостность — это обеспечение точности, непротиворечивости и корректности данных. В реляционной модели есть три основных вида целостности, которые реализуются через ограничения (constraints).

1.  **Целостность сущностей (Entity Integrity):**
    *   **Правило:** Первичный ключ (см. ниже) не может содержать `NULL`-значения.
    *   **Зачем:** Каждая строка в таблице должна быть уникально идентифицирована. Если идентификатор может быть пустым (`NULL`), то он ничего не идентифицирует, и уникальность нарушается.

2.  **Ссылочная целостность (Referential Integrity):**
    *   **Правило:** Значение внешнего ключа (см. ниже) должно либо совпадать со значением первичного ключа в связанной таблице, либо быть равным `NULL`.
    *   **Зачем:** Это правило предотвращает появление "осиротевших" записей. Нельзя создать заказ для клиента, которого не существует в базе. Нельзя удалить клиента, если на него ссылаются существующие заказы (если не заданы специальные правила, например, каскадное удаление).

3.  **Целостность доменов (Domain Integrity):**
    *   **Правило:** Все значения в столбце должны соответствовать его типу данных и заданным ограничениям (`CHECK`, `NOT NULL` и др.).
    *   **Зачем:** Обеспечивает корректность данных на самом базовом уровне. Нельзя вписать "Привет" в столбец с возрастом или отрицательное число в столбец с ценой товара.

### **Ключи отношения**

Ключи — это специальные атрибуты (или их наборы), которые используются для идентификации строк и установления связей.

#### **1. Потенциальный ключ (Candidate Key)**

**Потенциальный ключ** — это атрибут или минимальный набор атрибутов, который **уникально идентифицирует** каждый кортеж в отношении.

*   *Минимальный* означает, что если из набора убрать любой атрибут, он перестанет быть уникальным.
*   В одной таблице может быть несколько потенциальных ключей.

**Пример:** Таблица `Сотрудники`
(`ID`, `ТабельныйНомер`, `СерияНомерПаспорта`, `Имя`, `Фамилия`)

Здесь потенциальными ключами могут быть:
*   `{ID}` (уникальный идентификатор)
*   `{ТабельныйНомер}` (тоже уникален для каждого сотрудника)
*   `{СерияНомерПаспорта}` (также уникален)

#### **2. Первичный ключ (Primary Key, PK)**

**Первичный ключ** — это один из потенциальных ключей, выбранный разработчиком в качестве **основного** идентификатора для строк в таблице.

*   Для каждой таблицы должен быть только **один** первичный ключ.
*   Он должен быть **уникальным** и не может содержать `NULL` (это и есть правило целостности сущностей).
*   **Выбор:** Обычно выбирают самый простой, короткий и стабильный ключ (который никогда не меняется). Часто для этого создают искусственный суррогатный ключ (например, `ID` с автоинкрементом).

**Пример:** В таблице `Сотрудники` мы выбираем `{ID}` в качестве первичного ключа.

#### **3. Внешний ключ (Foreign Key, FK)**

**Внешний ключ** — это атрибут или набор атрибутов в одной таблице, который ссылается на **первичный ключ** в другой (или этой же) таблице.

*   **Назначение:** Устанавливает и обеспечивает логическую связь между двумя таблицами.
*   Внешний ключ — это механизм реализации **ссылочной целостности**.

**Пример:** У нас есть две таблицы:
1.  `Отделы` (`ID_Отдела` \[PK], `НазваниеОтдела`)
2.  `Сотрудники` (`ID` \[PK], `Имя`, `Фамилия`, `ID_Отдела_FK`)

В таблице `Сотрудники` поле `ID_Отдела_FK` является **внешним ключом**. Оно ссылается на первичный ключ `ID_Отдела` в таблице `Отделы`.

**Что это дает:**
*   Мы можем добавить в поле `ID_Отдела_FK` только те значения, которые уже существуют в столбце `ID_Отдела` таблицы `Отделы`.
*   Мы не можем создать сотрудника с `ID_Отдела_FK` = 5, если отдела с `ID_Отдела` = 5 не существует. Это и есть работа ссылочной целостности.

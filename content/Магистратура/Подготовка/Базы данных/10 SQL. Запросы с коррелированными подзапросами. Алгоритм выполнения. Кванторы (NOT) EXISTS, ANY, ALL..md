Отлично, мы переходим к одной из самых мощных и, порой, сложных тем в SQL — коррелированным подзапросам. Понимание их работы открывает двери к решению очень нетривиальных задач.

Продолжим использовать наши таблицы `Employees` и `Departments`.

---

### 1. Коррелированные подзапросы

#### В чем отличие от простого подзапроса?

*   **Простой (некоррелированный) подзапрос** — это независимый запрос. Он выполняется **один раз**, и его результат (одно значение, список или таблица) используется внешним запросом. Он ничего не знает о внешнем запросе.

*   **Коррелированный (связанный) подзапрос** — это зависимый запрос. Он **не может быть выполнен отдельно**, потому что он использует значения из **текущей строки**, обрабатываемой внешним запросом. Фактически, он выполняется заново **для каждой строки** внешнего запроса.

Можно представить это как вложенный цикл в программировании:

```
// Аналогия коррелированного подзапроса
for (row in outer_query_result) {
    // Внутренний подзапрос выполняется здесь, используя данные из 'row'
    result = execute_inner_query(row.value);
    if (condition_is_met(result)) {
        // Добавить 'row' в финальный результат
        add_to_final_result(row);
    }
}
```

#### Алгоритм выполнения коррелированного подзапроса

Давайте разберем на классическом примере.
**Задача:** Найти сотрудников, чья зарплата выше средней зарплаты в **их собственном** отделе.

```sql
SELECT
    e.first_name,
    e.salary,
    e.department_id
FROM
    Employees AS e
WHERE
    e.salary > (
        SELECT AVG(e_inner.salary)
        FROM Employees AS e_inner
        WHERE e_inner.department_id = e.department_id -- Корреляция!
    );
```

**Как SQL выполняет этот запрос:**

1.  **Внешний запрос начинает работу.** Он берет первую строку из таблицы `Employees`. Допустим, это Анна Сидорова (`id=4`, `salary=105000`, `department_id=20`).

2.  **Передача значения в подзапрос.** Для этой строки значение `e.department_id` (которое равно `20`) передается внутрь подзапроса.

3.  **Выполнение подзапроса.** Теперь подзапрос может быть выполнен. Он превращается в:
    `SELECT AVG(e_inner.salary) FROM Employees AS e_inner WHERE e_inner.department_id = 20;`
    СУБД вычисляет среднюю зарплату для отдела IT (id=20). Допустим, получилось `107500`.

4.  **Сравнение в `WHERE`.** Внешний запрос сравнивает зарплату Анны с результатом подзапроса: `WHERE 105000 > 107500`. Это ложь (`FALSE`).

5.  **Принятие решения.** Строка с Анной Сидоровой **не включается** в итоговый результат.

6.  **Переход к следующей строке.** Внешний запрос берет следующую строку, например, Петр Петров (`id=3`, `salary=120000`, `department_id=20`).

7.  **Повторение цикла.** Процесс повторяется. Значение `e.department_id` (снова `20`) передается в подзапрос. Подзапрос снова выполняется и возвращает `107500`.

8.  **Новое сравнение.** Внешний запрос делает сравнение: `WHERE 120000 > 107500`. Это истина (`TRUE`).

9.  **Принятие решения.** Строка с Петром Петровым **включается** в итоговый результат.

10. **И так далее...** Процесс повторяется для каждой строки таблицы `Employees`.

**Важное замечание о производительности:** Из-за того, что подзапрос выполняется для каждой строки, коррелированные подзапросы могут быть очень медленными на больших таблицах. Часто их можно (и нужно) заменить на `JOIN`. Например, предыдущую задачу можно решить так:

```sql
SELECT
    e.first_name,
    e.salary
FROM
    Employees AS e
JOIN
    (
        SELECT department_id, AVG(salary) as avg_dept_salary
        FROM Employees
        GROUP BY department_id
    ) AS dept_avg ON e.department_id = dept_avg.department_id
WHERE
    e.salary > dept_avg.avg_dept_salary;
```

### 2. Кванторы `EXISTS`, `ANY`, `ALL`

Эти операторы используются в `WHERE` или `HAVING` для сравнения значения с результатом подзапроса.

#### `[NOT] EXISTS`

Это самый частый и эффективный способ использования коррелированных подзапросов. `EXISTS` проверяет, вернул ли подзапрос **хотя бы одну строку**. Ему не важны значения в этих строках, только их наличие.

*   `EXISTS` возвращает `TRUE`, если подзапрос вернул 1 или более строк.
*   `NOT EXISTS` возвращает `TRUE`, если подзапрос вернул 0 строк (пустой набор).

**Пример `EXISTS`:** Найти все отделы, в которых **есть** хотя бы один сотрудник.

```sql
SELECT
    d.name
FROM
    Departments AS d
WHERE
    EXISTS (
        SELECT 1 -- Можно писать *, имя столбца, что угодно. Это не имеет значения.
        FROM Employees AS e
        WHERE e.department_id = d.id -- Корреляция
    );
```
**Алгоритм:** Для каждого отдела `d` SQL выполняет подзапрос, который ищет сотрудников с `department_id`, равным `d.id`. Если находит хотя бы одного — `EXISTS` истинно, и отдел попадает в результат.

**Пример `NOT EXISTS`:** Найти все отделы, в которых **нет** сотрудников.

```sql
SELECT
    d.name
FROM
    Departments AS d
WHERE
    NOT EXISTS (
        SELECT 1
        FROM Employees AS e
        WHERE e.department_id = d.id -- Корреляция
    );
```
**Алгоритм:** Для отдела "Финансы" (`id=40`) подзапрос не найдет ни одной строки в `Employees`. Подзапрос вернет пустой набор, и `NOT EXISTS` станет `TRUE`. Отдел "Финансы" попадет в результат.

#### `ANY` (или `SOME`)

Квантор `ANY` должен использоваться после оператора сравнения (`=`, `>`, `<`, и т.д.). Условие будет истинным, если сравнение верно **хотя бы для одного** значения из списка, возвращаемого подзапросом.

*   `= ANY` — эквивалентно `IN`.
*   `> ANY` — больше, чем минимальное значение в списке.
*   `< ANY` — меньше, чем максимальное значение в списке.

**Пример `> ANY`:** Найти сотрудников, чья зарплата **больше зарплаты хотя бы одного** сотрудника из отдела HR (id=30). (То есть больше минимальной зарплаты в HR).

```sql
-- В отделе HR (id=30) работает Елена Смирнова с зарплатой 75000.
SELECT first_name, salary
FROM Employees
WHERE salary > ANY (
    SELECT salary
    FROM Employees
    WHERE department_id = 30
);
-- Этот запрос найдет всех, у кого зарплата > 75000.
```

#### `ALL`

Квантор `ALL` также используется после оператора сравнения. Условие будет истинным, только если сравнение верно **для всех** значений из списка, возвращаемого подзапросом.

*   `> ALL` — больше, чем максимальное значение в списке.
*   `< ALL` — меньше, чем минимальное значение в списке.
*   `= ALL` — бессмысленно, если подзапрос возвращает больше одного уникального значения.

**Пример `> ALL`:** Найти сотрудников, чья зарплата **больше зарплаты каждого** сотрудника из отдела HR (id=30). (То есть больше максимальной зарплаты в HR).

```sql
-- В отделе HR (id=30) работает Елена Смирнова с зарплатой 75000.
SELECT first_name, salary
FROM Employees
WHERE salary > ALL (
    SELECT salary
    FROM Employees
    WHERE department_id = 30
);
-- Этот запрос тоже найдет всех, у кого зарплата > 75000,
-- так как в HR всего один сотрудник. Если бы их было несколько,
-- результат был бы другим.
```
**Пример поинтереснее:** Найти сотрудников, получающих больше, чем **любой** сотрудник из отдела продаж (id=10).
В отделе продаж зарплаты 130000 и 95000. Максимальная - 130000.

```sql
SELECT first_name, salary
FROM Employees
WHERE salary > ALL (
    SELECT salary
    FROM Employees
    WHERE department_id = 10
);
-- Этот запрос не вернет ни одной строки, т.к. ни у кого нет з/п > 130000.
```

### Когда что использовать: Краткая памятка

| Задача | Рекомендуемый подход | Почему? |
| :--- | :--- | :--- |
| Проверить, существует ли связанная запись (найти родителей, у которых есть дети). | `EXISTS` | Очень эффективно, так как СУБД останавливает поиск в подзапросе после первой же найденной строки. Не требует сравнения значений. |
| Проверить, что связанной записи не существует (найти родителей без детей). | `NOT EXISTS` | Эффективно и часто более читаемо, чем `LEFT JOIN ... WHERE key IS NULL`. |
| Сравнить значение с набором других значений (найти X, который больше/меньше всех/любого из Y). | `> ALL`, `< ANY` и т.д. | Позволяет выражать сложные логические условия. Часто можно заменить на подзапрос с `MIN()` или `MAX()`, что бывает понятнее. `... > ANY (subquery)` эквивалентно `... > (SELECT MIN(...) FROM subquery)`. |
| Проверить вхождение в список. | `IN` (или `= ANY`) | `IN` является стандартом де-факто для этой задачи и отлично читается. |

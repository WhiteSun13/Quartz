### Тема 5 – Эскизное и техническое проектирование

Итак, тема 5 – эскизное и техническое проектирование. На предыдущем этапе мы произвели анализ предметной области, данные этого этапа используются непосредственно для проектирования.

#### Этапы проектирования

Существует три этапа проектирования:
1.  **Эскизное проектирование**
2.  **Техническое проектирование**
3.  **Рабочее проектирование**

Как видите, у этапа проектирования есть **цикл обратной связи**, то есть этапы от следующего проектирования могут быть возвращены на предыдущий, для того чтобы изменить нашу идею. Бывают новые подробности, информация, и они приводят к тому, что система была спроектирована неправильно.

*   **Эскизное проектирование** включает в себя диаграммы процессов первого уровня.
*   **Технический проект** и **рабочий проект** — это комплекс уже готовых решений.

Потом уже как большинство результатов проектирования информационной системы можем предъявлять на предприятие какую-то готовую систему, решение для службы.

#### Основные задачи проектирования

Основной задачей проектирования является передача и взаимодействия предметов области в уровень проектирования информационной системы.

1.  **Определение критических участков.** Критический участок – это жизненно важный для нормального функционирования участок. Создание системы с точки зрения бизнеса, так и успешной реализации приемов проекта.
2.  **Сметные увеличения, сметные затраты, сроки внедрения, исследования обеспечения.**
3.  **Определение архитектуры, платформы организационной системы:**
    *   Тип архитектуры: файл-серверная или клиент-серверная.
        > **Лектор:** Почти такая старая вещь, согласны? Файл-серверная архитектура?
        > **Студент:** Ну, у нас, мне кажется, многие такого не знают.
        > **Лектор:** Ну, да, потому что с этим никто нас не сталкивал. Сейчас такое еще и…
    *   Количество уровней архитектуры.
    *   Степень централизации базы данных.
    *   Степень однородности базы данных.
    *   Одноименные сети.
    *   Требования к уровню сервисов сети.
    *   Оценка производительности системы.
    *   Требования безопасности доступа в служебную систему.
    *   Требования возможности использования внутрисистемных данных.

#### Требования безопасности доступа и обслуживания системы

Рекомендуемый перечень вопросов, которые нужно задать:
-   Каков уровень необходимой доступности системы для запросов пользователей?
-   Доступна ли вообще, и когда допустимы периоды профилактического простоя системы?
-   Когда допустимы периоды ограничения доступа к системе?
-   Какие данные после отказа системы нельзя получить из генератора, из других источников?
-   Объем повторно вводимой информации.
-   Каково допустимое время восстановления системы после сбоя?
-   Еще приложения, кроме информационной системы, работают на данном оборудовании?
-   Имеются ли резервные аппаратные средства на случай отказа основных?
-   Имеется ли запас возможностей оборудования, на котором функционирует информационная система?
-   Скорость передачи данных при резервном копировании?
-   Имеется ли специальный противопожароустойчивый носитель для хранения резервных копий?

### Жизненный цикл проектируемой системы (ГОСТ 34.601-90)

Жизненный цикл проектируемой системы имеет три этапа, создавался по ГОСТ 34.601-90.

#### 1. Эскизный проект

В него включают этапы разработки предварительных проектных решений по системе и ее частям, разработку документации. Эскизное проектирование проводится для сложных систем, не имеющих аналогов. Эскизный проект документирует описание предполагаемой системы.

На этапе эскизного проектирования определяются:
-   Функции системы.
-   Их цели.
-   Состав опыта, оборудования и отдельных задач.
-   Концепция, эксплуатационные базы и подробная структура.
-   Состав частей системы.
-   Функции и параметры основных программных средств.

**Документация эскизного проекта (ГОСТ 34.201-89):**
-   Пояснительная записка к эскизному проекту.
-   Общий уровень описания процесса деятельности.
-   Основные технические решения.
-   Мероприятия по подготовке объекта автоматизации к вводу в систему действий.

#### 2. Технический проект

Включает в себя разработку проектных решений по системе и ее частям, разработку документации на систему и ее части. В основном, цель — комплектование на основании технических требований и технических заданий на их разработку. Разработка заданий на проектирование в смежных частях проектов, объектах автоматизации.

Техническое проектирование – это проектирование архитектуры системы, включающее разработку структуры и интерфейсов компонентов, согласование функций технических и программных компонентов.

На данном этапе разрабатывают решения:
-   по функциональной структуре системы;
-   по функциям, например, организационной структуры;
-   по структуре технических средств;
-   по алгоритмам решения задач с применением языков;
-   по организации и ведению информационной базы;
-   по системе классификации и кодированию информации.

**Документация технического проектирования:**
Технический проект системы – это техническая документация, содержащая общесистемные проектные решения, локальные решения задач, а также оценку экономической эффективности информационной системы и перечень мероприятий по подготовке объекта к внедрению.

**Источники разработки технического проекта:**
-   Технические задачи.
-   Технические задания.
-   Эскизный проект.

#### Постановка задачи

Обоснование задачи — это описание экономических задач и их места в организации, подготовка исчерпывающих представлений о логике преобразования информации для получения результата. Обоснование задачи вводится в будущем в пользу информационной схемы владельцем бизнес-процесса в содружестве с разработчиками. Для постановки задачи используются сведения, необходимые и достаточные для полного представления и в логичной и в реформируемой экономной модели.

-   **Процессные модели** описывают последовательность преобразования материальных и информационных потоков компании по реализации производственных функций.
    -   **На верхнем уровне** модели системы отражается конкретность действий.
    -   **На втором уровне** функциональных систем отражаются технологически комбинированные бизнес-процессы, их взаимосвязи в виде основных направлений действий.
    -   **На третьем уровне** бизнес-процессы, это отдельные функции каждого направления действий, детализируются на отдельные бизнес-процессы.
    -   **Четвертый уровень** бизнес-процессов — это конкретные бизнес-процессы. Бизнес-функции описываются последовательностью элементарных технологических операций.
    -   **Мини-спецификации** — это описание элементарной операции, считается конкретной мини-спецификацией.

Вот пример одной из систем взаимодействия с поставщиками: бывает бизнес-процесс, бизнес-функции, ее технологические операции, ее спецификации. Дальше мы дадим пример как раз на основе взаимодействия с поставщиками для сети супермаркетов. Подробный пример проектирования, документирования, ввода-вывода, взаимодействия. Ну тоже совсем техническое. Но там, по-моему, вся эта презентация как раз-таки про это взаимодействие. Например, а там вообще есть дальше.

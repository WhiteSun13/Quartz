## 📌 Статическая маршрутизация — две основные задачи

> [!important]
> **Две основные задачи статической маршрутизации:**
> 1. **Организация связи внутри локальной сети** — чтобы компьютеры видели друг друга
> 2. **Выход за пределы локальной сети** — выход в интернет из локальной сети

Обе задачи используются до сих пор.

## 🔑 Маршрутизация в локальной сети — IPv4

### 📋 Принципы адресации в локальной сети

> [!info]
> **Три ключевых принципа:**
> 1. Любые два устройства могут **непосредственно** связываться друг с другом
> 2. Число адресов, выделенных для организации связи, **всегда составляет 2^n**, где n ≥ 2 (то есть 4, 8, 16, 32... — всегда кратно степени двойки)
> 3. Адресное пространство **сплошное** от и до, но **первый и последний адрес блока не используются** — они называются **network** и **broadcast** соответственно

### 📋 Основные переменные адресации (все в формате IPv4 — 32 бита)

| Переменная | Описание | Биты |
|-----------|----------|------|
| **Netmask (маска сети)** | Задаёт число компьютеров в локальной сети | Первые (32-n) бит = **единицы**, оставшиеся n бит = **нули** |
| **Network** | Адрес начала блока | Последние n бит всегда **равны нулю** |
| **Broadcast** | Адрес конца блока | Последние n бит всегда **равны единице** |
| **Host (конкретный хост)** | Адрес конкретной машины | Первые (32-n) бит — как у network/broadcast; последние n — конкретный адрес |

```
[Network]  [  ...хосты...  ]  [Broadcast]
  ↑                                ↑
 нули                           единицы
 (последние n бит)          (последние n бит)
```

## 🛠 Программная реализация — присвоение адресов

### 📋 Сравнение трёх платформ

| Платформа | Команда | Особенности |
|-----------|---------|-------------|
| **Linux (ifconfig)** | `ifconfig <интерфейс> <IP-адрес> <маска>` | Линейные команды, одноразовые |
| **Linux (пакет ip)** | `ip addr add <IP/маска> dev <интерфейс>` | Синтаксис как у Cisco; **древовидная структура** команд; де-факто стандарт сегодня |
| **Cisco** | Аналогично пакету ip Linux | Древовидная структура; переходите на интерфейс → присваиваете адрес |
| **Windows** | `ipconfig /all` | По-своему; `ipconfig` (не `ifconfig`); выводит полную таблицу маршрутизации для всех интерфейсов |

> [!note]
> У Linux сейчас у продвинутого — маршрутизация реализована **так же, как у Cisco**. Пакет `ip` стал де-факто стандартом.

> [!tip]
> **Отличие двух подходов в Linux:**
> - `ifconfig` — все команды **линейные**, в одной директории
> - `ip` — команды расположены **в дереве** (древовидная структура, как у Cisco)

### 📋 Добавление маршрутов (Linux)

```bash
# Вариант 1 — через route
route add -net <сеть> netmask <маска> gw <шлюз>

# Вариант 2 — через ip (древовидный интерфейс)
ip route add <сеть/маска> via <шлюз>
```

## 🌐 Выход за пределы локальной сети — Gateway (шлюз)

> [!important]
> **Шлюз (Gateway)** — специальный компьютер с **двумя интерфейсами**:
> - Один включён в **локальную сеть**
> - Второй — **внешний** (смотрит наружу)
>
> Все пакеты, отправляемые во внешнюю сеть, идут на внутренний интерфейс шлюза. Шлюз дальше разбирается, куда их перенаправить.

```
[PC1] ─┐
[PC2] ─┤── [Switch] ── [Gateway] ── [Интернет]
[PC3] ─┘      LAN        eth0/eth1
```

## ⚡ Динамическая маршрутизация

### 💡 Зачем нужна динамическая маршрутизация

> [!note]
> Магистральные каналы **нестабильны** — могут отключаться и пропадать. Для устойчивости обычно делают **несколько маршрутов** между узлами. Динамическая маршрутизация — это **автоматический выбор оптимального маршрута** из доступных.

**Пример:** От R1 до R3 можно пройти через R5, через R2 или более длинным путём — маршрутов несколько, нужно выбрать лучший.

### 📋 Два основных принципа динамической маршрутизации

1. **Порядок сбора информации о топологии сети** — какие узлы существуют, как соединены, состояние каналов связи
2. **Вычислительные процедуры для выбора маршрута** — должны быть **быстро сходящимися**, не зацикливаться

> [!tip]
> **Чем отличается Cisco:** Cisco изобрела и разработала процедуры с **быстрой сходимостью** — маршрут определяется очень быстро. Это одна из главных ценностей оборудования Cisco.

## 🔢 Метрическая функция — основа выбора маршрута

> [!important]
> **Метрическая функция** — ключевой элемент всех динамических протоколов. Служит для:
> - Оценки качества соединения
> - Выбора оптимального маршрута
>
> Для каждого участка маршрута определяется метрика → для составного маршрута из нескольких участков — складывается.
>
> **У каждого протокола — своя метрическая функция.**

### 🔑 Принцип наименьшего действия (из физики)

> [!info]
> Как и в физике (оптика, механика, квантовая механика — весь 10-томник Ландау и Лившица выводится из этого принципа), в маршрутизации действует **принцип наименьшего действия**:
>
> **Всегда выбирается тот маршрут, для которого значение метрической функции на его протяжении будет МИНИМАЛЬНЫМ.**

## 📊 Теоретические основы — теория графов

> [!info]
> Сеть — это **граф**:
> - **Вершины** = маршрутизаторы
> - **Рёбра** = каналы связи между маршрутизаторами
> - **Задача** = найти кратчайшее расстояние от одной вершины до другой
> - **Расстояние** = метрика (у каждого ребра своя длина)

### 📋 Два основных алгоритма

| Алгоритм | Применение |
|---------|-----------|
| **Алгоритм Беллмана-Форда** | Дистанционно-векторные протоколы (RIP) |
| **Алгоритм Дейкстры** | Протоколы состояния каналов (OSPF) |

> [!note]
> Вычислительная сложность алгоритма пропорциональна **времени на создание таблицы маршрутизации**. Хороший алгоритм — быстрый: задали граф с весами рёбер → таблица маршрутизации получена в кратчайшее время.

## 📡 Классификация протоколов маршрутизации

### 📋 По типу сети

| Тип | Описание |
|-----|----------|
| **Внутридоменная (IGP)** | Маршрутизация внутри одного провайдера; единые правила и центры управления |
| **Междудоменная (EGP)** | Глобальная сеть; взаимодействие между провайдерами; единица маршрутизации — **автономная система** |

> [!info]
> **Автономная система:** На глобальном уровне каждый провайдер — **одна точка, одна вершина графа**. Каждому провайдеру выдаётся **номер автономной системы (ASN)**.

## 🔧 Внутридоменные протоколы (IGP)

### 🔧 RIP — Routing Information Protocol

| Параметр | Значение |
|---------|---------|
| **Тип** | Дистанционно-векторный |
| **Версии** | RIP v1, RIP v2 |
| **Транспорт** | UDP, порт 520 |
| **Частота обновления** | каждые **30 секунд** — рассылаются все маршруты |
| **Метрика** | **Число переходов (hop count)** между узлами |
| **Принцип выбора** | Всегда выбирается маршрут с **наименьшим числом переходов** |
| **Максимум переходов** | **15** (16 = недостижимость) |
| **Алгоритм** | Беллмана-Форда |

> [!warning]
> **Недостаток RIP:** Метрика — только число переходов. RIP не учитывает пропускную способность канала. Может выбрать более короткий, но более узкий маршрут.

---

### 🔧 OSPF — Open Shortest Path First

| Параметр | Значение |
|---------|---------|
| **Стандарт** | RFC 2328 |
| **Тип** | Протокол состояния каналов (Link State) |
| **Назначение** | Маршрутизация внутри автономной системы |
| **Рассылка** | Только **обновления** маршрута (не вся таблица) |
| **Маски** | Поддерживает маски **переменной длины** |
| **Метрика** | **10^8 / B**, где B — доступная полоса пропускания (в кбит/с) |
| **Алгоритм** | **Дейкстры** |

> [!important]
> **Ключевое преимущество OSPF:** Выбирает не просто **кратчайший**, а **наиболее широкий** маршрут. Маршрут с большей пропускной способностью получает **меньшую метрику** → выбирается приоритетно.

**Пример сравнения RIP и OSPF:**

```
Маршрут A: 56 кбит/с (меньше переходов)
Маршрут B: толстые каналы (больше переходов)

RIP выберет → Маршрут A (меньше переходов)
OSPF выберет → Маршрут B (больше пропускная способность)
```

---

### 🔧 EIGRP — Enhanced Interior Gateway Routing Protocol

| Параметр | Значение |
|---------|---------|
| **Тип** | Проприетарный протокол **Cisco** (открыт с 2013 года) |
| **Особенность** | Быстрая сходимость |
| **Метрика** | Сложная формула, в простейшем случае: **Delay/10 + 10^7/MinBandwidth** |

**Формула метрики:**
```
Метрика = Delay (задержка / 10) + 10^7 / MinBandwidth (минимальная полоса пропускания)
```

> [!note]
> Формула сложнее, чем у OSPF, но инновация Cisco в том, что их роутеры считают её **практически мгновенно**.

### 📋 Сравнение внутридоменных протоколов

| Протокол | Тип | Метрика | Алгоритм | Особенность |
|---------|-----|---------|---------|------------|
| **RIP** | Дистанционно-векторный | Hop count (макс. 15) | Беллмана-Форда | Прост, но не учитывает пропускную способность |
| **OSPF** | Состояние каналов | 10^8 / B | Дейкстры | Выбирает широкие каналы |
| **EIGRP** | Гибридный (Cisco) | Delay + 10^7/BW | Диффузный | Быстрая сходимость; проприетарный Cisco |

## 🌍 Междудоменная маршрутизация — BGP

> [!important]
> **BGP (Border Gateway Protocol)** — **«король всех протоколов»**. Протокол внешнего шлюза для маршрутизации между автономными системами в глобальном интернете.
>
> **Кто настраивал BGP — тот настоящий интернет-администратор.** Дома настроить невозможно — нет автономной системы.

### 📋 Параметры BGP

| Параметр | Значение |
|---------|---------|
| **Версия** | **BGP-4** |
| **Стандарт** | RFC 1930 |
| **Тип** | Дистанционно-векторный (как RIP, но для магистрали) |
| **Транспорт** | **TCP**, порт **179** |
| **Метрика** | **Число переходов между автономными системами** |
| **Рассылка** | Доступность сетей между автономными системами |

> [!note]
> **Почему на магистрали вернулись к простой метрике (hop count)?** Для внутридоменных сетей городили огород с OSPF и EIGRP. Но на магистрали, где всё гораздо сложнее, решили **упростить** — используется число переходов между автономными системами.

### 📋 Автономная система (AS)

> [!info]
> **Автономная система** — минимальная независимая сеть с:
> - Единой политикой маршрутизации
> - Анонсируется на магистрали как **единое целое**
> - Имеет **уникальный номер ASN**

**История ASN:**
- Раньше: 16-битные номера (до ~65535)
- Раньше минимальный размер AS = **32 сети класса C**
- Сейчас минимальный размер AS = **1 сеть класса C**
- Сейчас ASN стали **шестизначными** (не пятизначными) — число автономных систем выросло многократно

**Рост таблицы BGP-маршрутизации:**

| Период | Размер оперативной памяти для таблицы |
|--------|--------------------------------------|
| ~1996–1997 | **8 МБ** (вся мировая таблица!) |
| Чуть позже | 16 МБ → 32 МБ → ... |
| Сейчас | **128–256 ГБ** |

> [!warning]
> **Одна из крупнейших проблем современного интернета** — размер таблицы BGP-маршрутизации вырос многократно из-за роста числа автономных систем. Каждый BGP-роутер на границе AS желательно должен хранить **полную таблицу** маршрутизации.

### 📋 Что нужно для настройки BGP

1. **Своё адресное пространство** (блок IP-адресов)
2. **Зарегистрированная автономная система** (номер ASN)
3. **LIR** (Local Internet Registry) — статус локального интернет-регистратора: самостоятельная регистрация своего адресного пространства в единой базе
4. **Актуальность базы WHOIS** — из ваших записей в базе вычерпывается вся ваша внутренняя инфраструктура
5. **Два внешних выхода** (для устойчивости — один упадёт, перейдёт на второй)

## 📡 Стандарты IEEE 802

> [!info]
> **IEEE 802** — группа стандартов, касающихся:
> - **ЛВС** (локальных вычислительных сетей)
> - **MAN** (городских сетей — Metropolitan Area Networks)
>
> Первый проект — **февраль 1980** года.
>
> Все спецификации **сертифицируются IEEE**.

### 📋 Таблица стандартов IEEE 802

| Стандарт | Описание | Актуальность |
|---------|----------|-------------|
| **802.1** | Управление сетевыми устройствами, их взаимодействие | ✅ Актуально |
| **802.2** | LLC (Logical Link Control) | ✅ |
| **802.3** | Технологии **Ethernet** | ✅ Сверхактуально |
| **802.4** | Token Bus | ❌ Не используется |
| **802.5** | Token Ring (MAC уровень) | ❌ |
| **802.6** | Городские сети (MAN) | ⚠️ |
| **802.7** | Широкополосная передача по коаксиальному кабелю | ❌ Не используется |
| **802.8** | Волоконно-оптические сети (FOLAN) | ✅ Актуально |
| **802.9** | Интегрированные сети (речь + данные) | ⚠️ Теряет актуальность |
| **802.10** | Сетевая безопасность | ✅ |
| **802.11** | **Беспроводные локальные сети (Wi-Fi)** | ✅ Сверхактуально |
| **802.12** | Кабельный модем | ✅ |
| **802.15** | **Wireless PAN (Bluetooth, UWB)** | ✅ |
| **802.16** | **WiMAX** (широкополосные беспроводные) | ✅ |
| **802.11s** | Самоорганизующиеся меш-сети | ✅ Актуально для беспилотников |

> [!note]
> Стандарты 802 описывают **два нижних уровня модели OSI** — физический и канальный.

## 🔗 Уровни модели OSI применительно к IEEE 802

### 📋 Разбивка канального уровня

```
Модель OSI:                IEEE 802:
┌──────────┐              ┌─────────────────────────┐
│ Канальный│ ──────────── │ LLC (Logical Link Control│  802.2
│  уровень │              │ — управление логической  │
│          │              │   связью)                │
│          │              ├─────────────────────────┤
│          │              │ MAC (Media Access Control│  802.3, 802.11...
│          │              │ — управление доступом    │
│          │              │   к среде передачи)      │
├──────────┤              └─────────────────────────┘
│Физический│ ──────────── Физические компоненты сети
│  уровень │
└──────────┘
```

> [!important]
> **MAC-адреса** — отсюда! MAC = Media Access Control.
> Канальный уровень делится на два подуровня:
> - **LLC** (Logical Link Control) — как устанавливается канал связи, управление логической связью
> - **MAC** (Media Access Control) — доступ к среде передачи

### 📋 Функции MAC-уровня

| Функция | Описание |
|---------|----------|
| **Управление доступом к среде передачи** | Совместный доступ к физическому уровню |
| **Определение границ кадров** | В IP-сетях — пакеты; в локальных сетях IEEE 802 — **кадры** |
| **Проверка ошибок кадров** | Контроль целостности |
| **Распознавание адресов кадров** | Идентификация получателя и отправителя |

> [!note]
> **TCP/IP vs OSI:** Протокол TCP/IP по уровням **полностью не совпадает** с моделью OSI. Это важно помнить.

> [!important]
> **Ключевые термины для запоминания:**
> - **Network** = первый адрес блока (все нули в хостовой части)
> - **Broadcast** = последний адрес блока (все единицы в хостовой части)
> - **Метрика** = функция оценки качества маршрута; **минимальная метрика = лучший маршрут**
> - **ASN** = номер автономной системы (для BGP)
> - **MAC** = Media Access Control (отсюда MAC-адреса)
> - **LLC** = Logical Link Control
> - **RIP** → hop count → Беллман-Форд
> - **OSPF** → 10^8/BW → Дейкстра → выбирает широкие каналы
> - **BGP** → TCP 179 → hop count между AS → «король протоколов»

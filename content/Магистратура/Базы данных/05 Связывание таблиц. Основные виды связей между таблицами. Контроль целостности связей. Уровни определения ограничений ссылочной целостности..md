Отлично! Мы разобрались с ключами, и теперь самое время посмотреть, как они работают вместе, чтобы создавать осмысленную структуру данных. Это и есть связывание таблиц.

### **1. Связывание таблиц: Зачем и Как?**

В реляционной модели мы стремимся избегать дублирования данных (этот процесс называется **нормализацией**). Вместо того чтобы хранить всю информацию о сущности и связанных с ней объектах в одной гигантской таблице, мы разбиваем ее на несколько логических таблиц.

**Зачем?**
*   **Устранение избыточности:** Информация об одном авторе хранится только в одном месте, даже если у него 100 книг.
*   **Обеспечение целостности:** Если нужно изменить имя автора, мы меняем его в одной строке, и это изменение автоматически отражается для всех его книг.
*   **Гибкость и масштабируемость:** Легче добавлять новые свойства (например, издательство для книг), не затрагивая другие таблицы.

**Как?**
Связывание таблиц осуществляется с помощью механизма **"первичный ключ – внешний ключ" (PK-FK)**.

*   В "главной" таблице (например, `Авторы`) есть **первичный ключ** (`ID_Автора`), который уникально идентифицирует каждую запись.
*   В "подчиненной" таблице (например, `Книги`) создается столбец (`ID_Автора_FK`), который является **внешним ключом** и хранит значение первичного ключа из таблицы `Авторы`.

Таким образом, для каждой книги мы точно знаем, кто ее автор.

### **2. Основные виды связей между таблицами**

Существует три основных типа связей, которые отражают логику взаимоотношений между сущностями реального мира.

#### **1) Связь "Один-ко-многим" (One-to-Many, 1:M)**

Это самый распространенный тип связи. Одна запись в таблице A может быть связана с **множеством** записей в таблице B, но одна запись в таблице B связана только с **одной** записью в таблице A.

*   **Пример:** `Авторы` и `Книги`.
    *   Один автор может написать много книг.
    *   Каждая конкретная книга написана только одним автором (в данном упрощенном примере).

*   **Реализация:**
    *   Таблица `Авторы` (сторона "один"): `ID_Автора` (PK), `ИмяАвтора`.
    *   Таблица `Книги` (сторона "многие"): `ID_Книги` (PK), `НазваниеКниги`, `**ID_Автора_FK**`.

    Внешний ключ (`ID_Автора_FK`) находится на стороне "многие".

#### **2) Связь "Один-к-одному" (One-to-One, 1:1)**

Одна запись в таблице A может быть связана только с **одной** записью в таблице B, и наоборот.

*   **Зачем это нужно?** Казалось бы, можно просто объединить их в одну таблицу. Причины для использования связи 1:1:
    1.  **Разделение по соображениям безопасности:** Хранить конфиденциальные данные (например, паспортные данные) отдельно от общедоступных (имя, должность).
    2.  **Выделение опциональных данных:** Если подробная информация есть не для всех записей (например, `Сотрудник` и `СлужебныйАвтомобиль`), ее выносят в отдельную таблицу, чтобы не создавать пустые столбцы в основной.
    3.  **Оптимизация производительности:** Разделение редко используемых данных от часто используемых.

*   **Пример:** `Сотрудники` и `РабочиеКонтакты`.
    *   У одного сотрудника есть только один набор рабочих контактов.
    *   Один набор рабочих контактов принадлежит только одному сотруднику.

*   **Реализация:**
    *   Таблица `Сотрудники`: `ID_Сотрудника` (PK), `Имя`.
    *   Таблица `РабочиеКонтакты`: `**ID_Сотрудника_FK**` (PK, FK), `РабочийТелефон`, `Email`.
    *   Ключевой момент: внешний ключ в подчиненной таблице (`РабочиеКонтакты`) одновременно является и ее **первичным ключом** (или имеет ограничение `UNIQUE`). Это гарантирует, что на одного сотрудника не получится завести два набора контактов.

#### **3) Связь "Многие-ко-многим" (Many-to-Many, M:N)**

Одна запись в таблице A может быть связана с **множеством** записей в таблице B, и одна запись в таблице B также может быть связана с **множеством** записей в таблице A.

*   **Пример:** `Студенты` и `Курсы`.
    *   Один студент может записаться на много курсов.
    *   На один курс может быть записано много студентов.

*   **Реализация:** Прямую связь M:N в реляционной модели создать нельзя. Она реализуется через **третью, связующую (или ассоциативную) таблицу**.

    1.  Таблица `Студенты`: `ID_Студента` (PK), `ИмяСтудента`.
    2.  Таблица `Курсы`: `ID_Курса` (PK), `НазваниеКурса`.
    3.  **Связующая таблица `ЗаписиНаКурс`**:
        *   `**ID_Студента_FK**` (FK, ссылается на `Студенты.ID_Студента`)
        *   `**ID_Курса_FK**` (FK, ссылается на `Курсы.ID_Курса`)

    Первичным ключом для этой связующей таблицы обычно является **составной ключ** из двух внешних ключей (`PK(ID_Студента_FK, ID_Курса_FK)`). Это гарантирует, что один и тот же студент не может записаться на один и тот же курс дважды.

### **3. Контроль целостности связей (Ссылочная целостность)**

**Ссылочная целостность** — это правило, которое гарантирует, что связи между таблицами остаются согласованными. СУБД следит за тем, чтобы не появилось "осиротевших" записей.

Например, нельзя:
*   Создать запись о книге, указав `ID_Автора`, которого не существует в таблице `Авторы`.
*   Удалить автора из таблицы `Авторы`, если в таблице `Книги` есть книги, которые на него ссылаются.

Это правило реализуется с помощью ограничения `FOREIGN KEY`.

```sql
CREATE TABLE Книги (
    ID_Книги INT PRIMARY KEY,
    НазваниеКниги VARCHAR(255),
    ID_Автора_FK INT,
    
    -- Определение ограничения ссылочной целостности
    FOREIGN KEY (ID_Автора_FK) REFERENCES Авторы(ID_Автора)
);
```

### **4. Уровни определения ограничений ссылочной целостности**

Что должно произойти с дочерними записями (например, с книгами), если родительская запись (автор) **удаляется** или ее первичный ключ **изменяется**? Для этого в SQL при определении внешнего ключа можно указать специальные действия: `ON DELETE` и `ON UPDATE`.

#### **Действия `ON DELETE` (при удалении родительской записи):**

1.  **`RESTRICT` (ограничить) / `NO ACTION` (без действия)**
    *   **Что делает:** Запрещает удаление родительской записи, если на нее есть ссылки из дочерней таблицы. Это поведение по умолчанию в большинстве СУБД.
    *   **Пример:** Нельзя удалить автора, если у него есть книги. Чтобы удалить автора, нужно сначала удалить все его книги.
    *   *Примечание:* `NO ACTION` и `RESTRICT` почти идентичны. Разница в моменте проверки: `RESTRICT` проверяет сразу, `NO ACTION` — в конце транзакции.

2.  **`CASCADE` (каскадное удаление)**
    *   **Что делает:** При удалении родительской записи **автоматически удаляются все связанные с ней дочерние записи**.
    *   **Пример:** При удалении заказа (`Заказы`) каскадно удаляются все позиции этого заказа (`ПозицииЗаказа`). Это логично, так как позиции не могут существовать без заказа.
    *   **Осторожно!** Использовать с умом, так как можно случайно удалить большой объем данных.

3.  **`SET NULL` (установить NULL)**
    *   **Что делает:** При удалении родительской записи в поле внешнего ключа во всех связанных дочерних записях устанавливается значение `NULL`.
    *   **Пример:** Есть таблица `Сотрудники` и `Проекты`, где у каждого проекта есть поле `Руководитель_FK`. Если сотрудника-руководителя увольняют (удаляют из `Сотрудники`), то проект не удаляется, а просто становится "без руководителя" (`Руководитель_FK` = `NULL`).
    *   **Требование:** Поле внешнего ключа должно позволять `NULL`-значения.

#### **Действия `ON UPDATE` (при изменении PK родительской записи):**

Изменение первичного ключа — редкая операция, но возможная. Правила здесь аналогичны.

1.  **`RESTRICT` / `NO ACTION`:** Запретить изменение PK родительской записи, если есть ссылки.
2.  **`CASCADE`:** При изменении PK в родительской таблице это новое значение **автоматически прописывается во все внешние ключи** в дочерних таблицах. Очень удобно, если нужно изменить идентификатор.
3.  **`SET NULL`:** При изменении PK родителя внешний ключ в дочерних записях становится `NULL`.

**Полный синтаксис:**

```sql
CREATE TABLE Книги (
    ID_Книги INT PRIMARY KEY,
    НазваниеКниги VARCHAR(255),
    ID_Автора_FK INT,
    
    FOREIGN KEY (ID_Автора_FK) REFERENCES Авторы(ID_Автора)
        ON DELETE RESTRICT   -- Запретить удаление автора, если у него есть книги
        ON UPDATE CASCADE;   -- Если ID автора изменится, обновить его и для всех его книг
);
```

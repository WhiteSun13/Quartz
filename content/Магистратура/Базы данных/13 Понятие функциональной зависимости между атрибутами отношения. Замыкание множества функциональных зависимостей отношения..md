Понятие функциональной зависимости между атрибутами отношения. Замыкание множества функциональных зависимостей отношения. Неприводимое множество функциональных зависимостей. Тривиальная функциональная зависимость. Транзитивная функциональная зависимость. Зависимости, определяющие избыточность данных в отношении. Декомпозиция без потерь.

Это очень глубокая и теоретическая тема, лежащая в основе всего реляционного проектирования. Давайте разложим эти понятия по полочкам, чтобы они стали интуитивно понятны.

### 1. Понятие функциональной зависимости (ФЗ)

**Функциональная зависимость (Functional Dependency, FD)** — это центральное понятие. Это ограничение между двумя множествами атрибутов (столбцов) в отношении (таблице).

**Определение:** Говорят, что в отношении R существует функциональная зависимость **X → Y** (читается "X функционально определяет Y"), если для любых двух кортежей (строк) `t1` и `t2` из R, у которых значения атрибутов X совпадают, значения атрибутов Y также должны совпадать.

**Простыми словами:** Если мы знаем значение(я) в столбце(ах) X, мы можем **однозначно** определить значение(я) в столбце(ах) Y.

**Классический пример:**

Таблица `Employees`

| EmployeeID | FullName | DepartmentID | DepartmentName |
| :--- | :--- | :--- | :--- |
| 101 | Иван Иванов | 5 | Продажи |
| 102 | Петр Петров | 5 | Продажи |
| 103 | Анна Сидорова | 7 | IT |

Здесь верны следующие ФЗ:
*   `EmployeeID → FullName` (Зная ID, мы точно знаем ФИО)
*   `EmployeeID → DepartmentID` (Зная ID, мы точно знаем ID его отдела)
*   `DepartmentID → DepartmentName` (Зная ID отдела, мы точно знаем его название)
*   `(EmployeeID, FullName) → DepartmentID` (Эта ФЗ тоже верна, но `FullName` здесь избыточен)

А вот `DepartmentID → EmployeeID` **неверно**, потому что значению `DepartmentID = 5` соответствуют два разных `EmployeeID` (101 и 102).

### 2. Типы функциональных зависимостей

#### Тривиальная функциональная зависимость
**Определение:** ФЗ `X → Y` является **тривиальной**, если Y является подмножеством X.
*   **Пример:** `(EmployeeID, FullName) → FullName`.
*   **Смысл:** Такая зависимость не несет никакой новой информации. Очевидно, что если мы знаем ID и ФИО, то мы знаем и ФИО. Они бесполезны для анализа и нормализации.

#### Транзитивная функциональная зависимость
**Определение:** Пусть есть атрибуты X, Y, Z. Если существуют зависимости **X → Y** и **Y → Z**, но при этом **Y → X** неверно, то зависимость **X → Z** называется **транзитивной**.
*   **Пример из таблицы выше:**
    *   `EmployeeID → DepartmentID`
    *   `DepartmentID → DepartmentName`
    *   Следовательно, `EmployeeID → DepartmentName` является **транзитивной** зависимостью.
*   **Смысл:** Это зависимость "через посредника". `EmployeeID` определяет `DepartmentName` не напрямую, а через `DepartmentID`. **Именно транзитивные зависимости приводят к аномалиям и устраняются при переходе к третьей нормальной форме (3НФ).**

---

### 3. Зависимости, определяющие избыточность данных

Избыточность — это повторение одной и той же информации в нескольких местах. Она порождается определенными типами ФЗ:

1.  **Транзитивные зависимости:** В нашем примере `DepartmentName` "Продажи" повторяется для каждого сотрудника этого отдела. Это и есть избыточность, порожденная зависимостью `EmployeeID → DepartmentName`.
2.  **Частичные зависимости (для составных ключей):** Если атрибут зависит не от всего составного ключа, а только от его части. Это проблема, которую решает 2НФ.

---

### 4. Замыкание множества функциональных зависимостей (F+)

Представьте, что у нас есть исходный набор ФЗ `F`, который мы выявили при анализе предметной области. Но из этих базовых зависимостей можно логически вывести множество других.

**Замыкание множества ФЗ (F+)** — это полное множество всех функциональных зависимостей, которые можно логически вывести из исходного множества F.

Для вывода используются **аксиомы Армстронга**:
1.  **Рефлексивность:** Если Y ⊆ X, то X → Y (основа для тривиальных ФЗ).
2.  **Дополнение (Augmentation):** Если X → Y, то XZ → YZ (если мы знаем Y по X, то, добавив Z к обеим частям, зависимость сохранится).
3.  **Транзитивность:** Если X → Y и Y → Z, то X → Z (уже знакомое нам правило).

**Пример:**
*   Дано F = { A → B, B → C }
*   Тогда F+ будет содержать:
    *   A → B (из F)
    *   B → C (из F)
    *   A → C (по транзитивности)
    *   A → A, B → B, C → C, ... (тривиальные по рефлексивности)
    *   AC → BC, AB → CB (по дополнению)
    *   ... и много других.

**Зачем это нужно?** Замыкание — это мощный теоретический инструмент, чтобы, например, проверить, эквивалентны ли два разных набора ФЗ. На практике чаще используется замыкание множества атрибутов.

---

### 5. Неприводимое (минимальное) множество функциональных зависимостей

Замыкание F+ обычно огромно. Нам же нужен **минимальный** набор ФЗ, который эквивалентен исходному (т.е. имеет то же самое замыкание). Этот набор называется **неприводимым множеством** или **каноническим покрытием**.

**Множество ФЗ F является неприводимым, если:**
1.  **Правая часть атомарна:** Все зависимости в F имеют вид `X → A`, где A — один атрибут.
2.  **Левая часть не избыточна:** Нельзя удалить ни один атрибут из левой части любой зависимости без изменения замыкания F+.
3.  **Нет лишних зависимостей:** Нельзя удалить ни одну зависимость из F без изменения замыкания F+.

**Зачем это нужно?** Это "очищенный" от всего мусора набор правил нашей предметной области. Он используется в алгоритмах синтеза схем БД (например, для достижения 3НФ).

### 6. Декомпозиция без потерь (Lossless-Join Decomposition)

Когда мы находим "плохие" зависимости (транзитивные, частичные), мы выполняем **декомпозицию** — разбиваем одну большую таблицу на несколько маленьких.

**Декомпозиция без потерь** — это такое разбиение, при котором, соединив (`JOIN`) полученные таблицы обратно, мы получим **в точности исходную таблицу**, без лишних или потерянных строк.

**Ключевое правило (Теорема Хита):**
Декомпозиция отношения R на два отношения R1 и R2 является декомпозицией без потерь тогда и только тогда, когда:
1.  Множество атрибутов R1, объединенное с множеством атрибутов R2, должно быть равно множеству атрибутов R. (Мы не потеряли столбцы).
2.  Пересечение множеств атрибутов R1 и R2 (**общие столбцы**) должно быть **суперключом** для хотя бы одного из отношений R1 или R2.

**Пример:**
Возьмем нашу таблицу `Employees_Bad(EmployeeID, FullName, DepartmentID, DepartmentName)`.
В ней есть транзитивная зависимость `EmployeeID → DepartmentID → DepartmentName`.

Мы выполняем декомпозицию на:
*   `R1 (EmployeeID, FullName, DepartmentID)`
*   `R2 (DepartmentID, DepartmentName)`

Проверим по правилу:
1.  Атрибуты R1 ∪ Атрибуты R2 = все исходные атрибуты. **Верно.**
2.  Пересечение атрибутов — это `{DepartmentID}`.
3.  Является ли `{DepartmentID}` суперключом для R1 или R2?
    *   Для R1 (`Employees`) — нет.
    *   Для R2 (`Departments`) — **да**, так как `DepartmentID → DepartmentName` является ФЗ, и `DepartmentID` — это первичный ключ R2.

**Вывод:** Поскольку пересечение является ключом для R2, эта декомпозиция является **декомпозицией без потерь**. Мы можем без проблем соединить эти две таблицы по `DepartmentID` и получить исходные данные.

Если бы мы разбили таблицу на `R1(EmployeeID, DepartmentName)` и `R2(FullName, DepartmentID)`, то при соединении по `DepartmentID` мы бы получили "смесь" из всех имен и всех названий отделов — это была бы декомпозиция с потерями (а точнее, с появлением ложных данных).

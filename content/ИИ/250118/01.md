## Цель работы

Целью данной работы является освоение инструментария системной инженерии MATLAB System Composer и применение его для проектирования архитектурной модели информационной системы электронной библиотеки университета. Работа демонстрирует процесс создания распределённой клиент-серверной архитектуры с интеграцией внешних сервисов, позволяющей обеспечить масштабируемость, отказоустойчивость и соответствие функциональным требованиям современной библиотечной системы.

---

## Теоретические сведения

Системная инженерия — это междисциплинарный подход, направленный на создание и управление сложными системами на протяжении всего их жизненного цикла. Ключевым аспектом является формальное описание архитектуры системы, которое включает её структуру (компоненты), поведение и взаимосвязи. Такой подход позволяет выявить противоречия, оценить выполнимость требований и снизить риски на ранних этапах проектирования.

MATLAB System Composer — это среда визуального моделирования, предназначенная для проектирования и анализа архитектуры систем. Она позволяет создавать иерархические модели, определять компоненты и их интерфейсы (порты), устанавливать связи (коннекторы), а также привязывать и отслеживать требования к элементам модели. System Composer поддерживает различные архитектурные парадигмы и часто используется в областях, где важна надёжность, безопасность и соответствие стандартам.

Архитектура современных библиотечных информационных систем строится на принципах микросервисной архитектуры, обеспечивающей независимое масштабирование компонентов, использование специализированных хранилищ данных и асинхронное взаимодействие через очереди сообщений.

---

## Задача

Спроектировать и создать в среде MATLAB System Composer иерархическую архитектурную модель информационной системы электронной библиотеки университета, включающую клиентскую и серверную подсистемы, интеграцию с внешними сервисами (LDAP, платёжный шлюз, LMS), определение интерфейсов обмена данными, а также формирование и привязку функциональных требований к элементам модели.

## Ход работы

### 1. Проектирование архитектуры системы

На первом этапе была разработана архитектура верхнего уровня информационной системы электронной библиотеки, представленная десятью основными компонентами (Рисунок 1).

**Компонент Client** (Клиентское приложение) представляет собой мобильное или веб-приложение для конечных пользователей. Он обеспечивает аутентификацию пользователей, поиск и просмотр книг, управление закладками и синхронизацию данных между устройствами. Компонент поддерживает офлайн-режим работы благодаря локальному кэшированию данных.

**Компонент Server** (Серверная подсистема) является центральным вычислительным ядром системы, реализующим бизнес-логику библиотеки. Включает сервисы аутентификации, поиска, доставки контента, бронирования, аналитики и уведомлений. Использует специализированные хранилища данных: PostgreSQL для транзакционных операций, MongoDB для документов и профилей, Elasticsearch для полнотекстового поиска, Redis для кэширования.

**Компонент UniversityLDAP** (Университетский LDAP-сервер) обеспечивает централизованную аутентификацию пользователей с использованием корпоративных учётных записей университета. Взаимодействует с сервером через интерфейсы IUserCredentials и IUserProfile.

**Компонент PaymentGateway** (Платёжный шлюз) интегрирует систему с внешними платёжными сервисами для обработки оплаты штрафов, продления сроков бронирования и приобретения электронных копий книг.

**Компонент EmailService** (Сервис электронной почты) осуществляет отправку email-уведомлений пользователям о событиях системы: напоминания о сроках возврата, подтверждения бронирования, информация о новых поступлениях.

**Компонент PushNotificationService** (Сервис push-уведомлений) обеспечивает доставку мгновенных уведомлений на мобильные устройства пользователей через платформы Apple Push Notification Service и Firebase Cloud Messaging.

**Компонент ISBNDatabase** (База данных ISBN) предоставляет внешний API для получения расширенных метаданных книг по международному стандартному книжному номеру.

**Компонент LMSIntegration** (Интеграция с LMS) обеспечивает двунаправленный обмен данными с системой управления обучением университета для синхронизации учебных материалов и отслеживания прогресса студентов.

**Компонент MonitoringInfrastructure** (Инфраструктура мониторинга) осуществляет наблюдение за состоянием всех сервисов системы, сбор метрик производительности и генерацию оповещений при возникновении инцидентов.

### 2. Определение интерфейсов и связей между компонентами

Для обеспечения взаимодействия между компонентами были определены типизированные интерфейсы данных и установлены соединители:

**Интерфейсы аутентификации:**
- **IUserCredentials** — содержит поля username, password, authMethod, timestamp для передачи учётных данных
- **IAuthToken** — содержит accessToken, refreshToken, expiresIn, userRole, userId для управления сессиями
- **IUserProfile** — содержит userId, fullName, email, faculty, group, borrowedBooksCount, maxBorrowLimit

**Интерфейсы работы с контентом:**
- **IBookContentRequest** — содержит bookId, format, startPage, endPage для запроса контента
- **IBookContent** — содержит bookId, data, totalSize, checksum для передачи контента
- **IBookMetadata** — содержит bookId, title, authors, isbn, coverUrl, accessRules для метаданных
- **ISearchQuery** — содержит queryText, searchFields, page, pageSize для поисковых запросов
- **ISearchResults** — содержит totalCount, books (массив IBookMetadata), searchTime_ms

**Интерфейсы синхронизации и кэширования:**
- **ICacheData** — содержит key, value, ttl, operation для операций с кэшем

**Интерфейсы бронирования и уведомлений:**
- **IBooking** — содержит bookingId, userId, bookId, dueDate, status
- **INotification** — содержит type, title, message, priority

**Интерфейс мониторинга:**
- **ISystemMetrics** — содержит serviceName, cpuUsage, requestCount, avgResponseTime_ms

**Основные связи между компонентами верхнего уровня:**
1) Client ↔ Server — аутентификация, поиск, доставка контента, синхронизация данных
2) Server ↔ UniversityLDAP — верификация учётных данных, получение профилей
3) Server ↔ PaymentGateway — обработка платежей за бронирование
4) Server → EmailService — отправка email-уведомлений
5) Server → PushNotificationService — отправка push-уведомлений
6) Server ↔ ISBNDatabase — запрос метаданных книг
7) Server ↔ LMSIntegration — синхронизация с системой обучения
8) MonitoringInfrastructure ↔ Server — мониторинг состояния и сбор метрик

### 3. Детализация внутренней структуры компонентов

#### 3.1 Детализация компонента Client

Компонент Client был детализирован до уровня девяти подкомпонентов:

**AuthenticationClient** управляет процессом аутентификации пользователя, хранением токенов и их передачей другим модулям клиента. Порты: CredentialsOut (IUserCredentials), TokenIn (IAuthToken), TokenOut (IAuthToken).

**SearchEngineClient** формирует поисковые запросы и обрабатывает результаты поиска для отображения пользователю. Порты: QueryOut (ISearchQuery), ResultsIn (ISearchResults).

**BookViewer** обеспечивает отображение контента книги, навигацию по страницам и фиксацию прогресса чтения. Порты: ContentRequestOut (IBookContentRequest), ContentIn (IBookContent), BookmarkOut (ICacheData), ProgressOut (IAnalyticsEvent).

**DownloadManager** координирует загрузку контента книг с сервера и сохранение в локальное хранилище. Порты: ViewerRequestIn (IBookContentRequest), RequestOut (IBookContentRequest), ExternalContentIn (IBookContent), ToStorageOut (IBookContent).

**OfflineStorage** обеспечивает персистентное хранение загруженных книг для офлайн-доступа. Порты: StoreIn (IBookContent), RetrieveOut (IBookContent).

**BookmarkManager** управляет созданием, редактированием и синхронизацией закладок пользователя. Порты: BookmarkDataIn (ICacheData), SyncIn (ICacheData), SyncOut (IAnalyticsEvent).

**ReadingProgressTracker** отслеживает прогресс чтения и формирует аналитические события. Порты: ProgressIn (IAnalyticsEvent), AnalyticsOut (IAnalyticsEvent), SyncDataOut (ICacheData).

**LocalCache** обеспечивает быстрый доступ к часто используемым данным и буферизацию изменений для синхронизации. Порты: CacheRequestIn (ICacheData), CacheResponseOut (ICacheData), ToSyncEngineOut (ICacheData), UpdateFromSyncIn (ICacheData).

**SyncEngine** координирует двунаправленную синхронизацию данных между клиентом и сервером. Порты: LocalChangesIn (ICacheData), LocalUpdateOut (ICacheData), ProgressDataIn (ICacheData), TokenIn (IAuthToken), RemoteSyncIn (ICacheData), RemoteSyncOut (ICacheData).

*Рис. 2. Детализация компонента Client*

#### 3.2 Детализация компонента Server

Компонент Server был детализирован до уровня семнадцати подкомпонентов:

**AuthenticationService** обрабатывает запросы аутентификации, взаимодействует с LDAP и PostgreSQL для верификации пользователей.

**SearchService** обрабатывает поисковые запросы, взаимодействует с ElasticsearchCluster для полнотекстового поиска.

**ContentDeliveryService** обеспечивает доставку контента книг с использованием кэширования через Redis и получением данных из ObjectStorage.

**CatalogService** управляет каталогом книг, хранит метаданные в MongoDB, интегрируется с ISBNDatabase.

**BookingService** обрабатывает бронирование книг, интегрируется с PaymentGateway и отправляет уведомления.

**UserService** управляет профилями пользователей, синхронизирует данные клиентов, интегрируется с LMS.

**NotificationService** координирует отправку уведомлений через Email и Push-сервисы.

**AnalyticsService** собирает аналитические события и передаёт их в Kafka для потоковой обработки.

**HealthMonitor** отслеживает состояние сервисов и предоставляет метрики инфраструктуре мониторинга.

**Компоненты хранения данных:**
- **PostgreSQLCluster** — реляционная БД для пользователей и бронирований
- **MongoDBCluster** — документная БД для каталога и профилей
- **ElasticsearchCluster** — поисковый движок
- **RedisCluster** — распределённый кэш
- **ObjectStorage** — хранилище файлов контента книг
- **KafkaCluster** — брокер сообщений для аналитики
- **ClickHouseCluster** — колоночная БД для аналитических данных
- **RabbitMQCluster** — очередь сообщений для уведомлений

### 4. Формирование и привязка функциональных требований

На следующем этапе работы были сформулированы функциональные требования к системе, организованные в иерархическую структуру. Было создано шестнадцать групп требований, охватывающих все аспекты работы системы.

**REQ-AUTH (Аутентификация)** — требования к аутентификации через LDAP, генерации JWT-токенов, управлению сессиями.
- REQ-AUTH-001: Аутентификация через университетский LDAP с использованием IUserCredentials
- REQ-AUTH-002: Генерация accessToken и refreshToken согласно IAuthToken
- REQ-AUTH-003: Время жизни токена не менее 3600 секунд
- REQ-AUTH-004: Передача токена в SyncEngine для авторизации синхронизации

**REQ-SEARCH (Поиск)** — требования к полнотекстовому поиску и обработке результатов.
- REQ-SEARCH-001: Приём запросов через ISearchQuery, возврат через ISearchResults
- REQ-SEARCH-002: Полнотекстовый поиск в Elasticsearch по queryText
- REQ-SEARCH-003: Поддержка пагинации через page и pageSize
- REQ-SEARCH-004: Возврат метаданных книг в формате IBookMetadata

**REQ-CONTENT (Контент)** — требования к доставке и кэшированию контента книг.
- REQ-CONTENT-001: Запрос контента с указанием bookId, формата и диапазона страниц
- REQ-CONTENT-002: Проверка Redis перед обращением к ObjectStorage
- REQ-CONTENT-003: Проверка целостности через checksum
- REQ-CONTENT-004: Сохранение в OfflineStorage для офлайн-доступа
- REQ-CONTENT-005: Чтение из OfflineStorage при отсутствии сети

**REQ-BOOKMARK-PROGRESS (Закладки и прогресс)** — требования к управлению закладками и отслеживанию чтения.
- REQ-BOOKMARK-001: Передача закладок через ICacheData
- REQ-BOOKMARK-002: Кэширование закладок в LocalCache
- REQ-PROGRESS-001: Отправка событий прогресса через IAnalyticsEvent
- REQ-PROGRESS-002: Передача прогресса в SyncEngine

**REQ-SYNC (Синхронизация)** — требования к синхронизации данных клиент-сервер.
- REQ-SYNC-001: Отправка локальных изменений через RemoteSyncOut
- REQ-SYNC-002: Получение обновлений через RemoteSyncIn
- REQ-SYNC-003: Авторизация синхронизации через IAuthToken

**REQ-BOOKING (Бронирование)** — требования к бронированию физических копий.
- REQ-BOOKING-001: Обработка бронирования с сохранением в PostgreSQL
- REQ-BOOKING-002: Интеграция с PaymentGateway через IBooking
- REQ-BOOKING-003: Отправка уведомлений через RabbitMQ

**REQ-NOTIFY (Уведомления)** — требования к системе уведомлений.
- REQ-NOTIFY-001: Получение задач из RabbitMQ через INotification
- REQ-NOTIFY-002: Отправка email через EmailService
- REQ-NOTIFY-003: Отправка push через PushNotificationService
- REQ-NOTIFY-004: Публикация уведомлений о новых книгах

**REQ-CATALOG (Каталог)** — требования к управлению каталогом.
- REQ-CATALOG-001: Хранение метаданных в MongoDB
- REQ-CATALOG-002: Интеграция с ISBNDatabase

**REQ-ANALYTICS (Аналитика)** — требования к сбору и обработке аналитики.
- REQ-ANALYTICS-001: Сбор событий через IAnalyticsEvent
- REQ-ANALYTICS-002: Потоковая обработка через Kafka
- REQ-ANALYTICS-003: Хранение в ClickHouse

**REQ-LMS (LMS интеграция)** — требования к интеграции с системой обучения.
- REQ-LMS-001: Получение данных курсов через LMSDataIn
- REQ-LMS-002: Отправка прогресса через LMSDataOut

**REQ-USER (Пользователи)** — требования к управлению профилями.
- REQ-USER-001: Хранение профилей в MongoDB
- REQ-USER-002: Синхронизация через ICacheData

**REQ-PERF (Производительность)** — требования к производительности.
- REQ-PERF-001: searchTime_ms не более 500 мс для 95% запросов
- REQ-PERF-002: Время доступа к Redis менее 10 мс
- REQ-PERF-003: Управление TTL через поле ttl в ICacheData

**REQ-MONITOR (Мониторинг)** — требования к мониторингу системы.
- REQ-MONITOR-001: Сбор метрик через ISystemMetrics
- REQ-MONITOR-002: Периодические health-check запросы
- REQ-MONITOR-003: Отправка метрик в MonitoringInfrastructure

**REQ-SEC (Безопасность)** — требования к безопасности.
- REQ-SEC-001: Шифрование пароля в IUserCredentials
- REQ-SEC-002: Ролевая модель через userRole
- REQ-SEC-003: Правила доступа через accessRules

**REQ-REL (Надёжность)** — требования к надёжности.
- REQ-REL-001: Офлайн-режим через LocalCache и OfflineStorage
- REQ-REL-002: Сохранение изменений при недоступности сервера

**REQ-PROFILE (Данные пользователя)** — требования к профилю.
- REQ-PROFILE-001: Полный набор полей IUserProfile
- REQ-PROFILE-002: Контроль borrowedBooksCount ≤ maxBorrowLimit

*Рис. 4. Иерархическая структура функциональных требований*

С использованием возможностей System Composer все требования были явно привязаны к соответствующим компонентам и подкомпонентам архитектурной модели.

В ходе работы успешно спроектирована архитектурная модель информационной системы электронной библиотеки университета в среде MATLAB System Composer. Была создана иерархическая модель, включающая десять компонентов верхнего уровня (Client, Server, UniversityLDAP, PaymentGateway, EmailService, PushNotificationService, ISBNDatabase, LMSIntegration, MonitoringInfrastructure) с детализацией клиентской подсистемы до девяти подкомпонентов и серверной — до семнадцати.

Сформулировано шестьдесят функциональных требований, организованных в шестнадцать тематических групп, охватывающих аутентификацию, поиск, доставку контента, синхронизацию, бронирование, уведомления, аналитику, интеграции, производительность, безопасность и надёжность системы.

Разработанная архитектура демонстрирует применение современных паттернов проектирования: микросервисная архитектура серверной части, использование специализированных хранилищ данных (PostgreSQL, MongoDB, Elasticsearch, Redis, ClickHouse), асинхронное взаимодействие через очереди сообщений (Kafka, RabbitMQ), поддержка офлайн-режима на клиенте. Результаты работы могут служить основой для дальнейшей реализации, тестирования и сопровождения системы.

Вот проект четвертой главы для лабораторной работы. Она базируется на массиве `Requirements` из предоставленного файла и описывает, как функциональные и нефункциональные требования были структурированы и привязаны к архитектурным элементам системы.

### 4. Формирование и привязка функциональных требований

В рамках проектирования системы **ELIBRARYSYSTEM** был разработан реестр требований, структурированный по функциональным доменам. Каждое требование имеет уникальный идентификатор (ID), приоритет и описание. Для обеспечения сквозной трассируемости (traceability) требования были явно привязаны к реализующим их компонентам и интерфейсам.

#### 4.1. Структура реестра требований

Требования организованы иерархически с использованием контейнеров для группировки по подсистемам. Всего выделено 16 функциональных групп.

**4.1.1. Подсистема безопасности и доступа (REQ-AUTH, REQ-SEC)**
Ключевые требования касаются интеграции с корпоративными системами и защиты данных:
*   **REQ-AUTH-001:** Реализация аутентификации через внешний LDAP-сервер (UniversityLDAP).
*   **REQ-AUTH-002/003:** Генерация и управление жизненным циклом JWT-токенов (Access/Refresh) через интерфейс `IAuthToken`.
*   **REQ-SEC-001:** Обязательное шифрование поля `password` в интерфейсе `IUserCredentials`.
*   **REQ-SEC-002:** Реализация ролевой модели доступа (RBAC) на основе поля `userRole`.

**4.1.2. Поиск и Каталог (REQ-SEARCH, REQ-CATALOG)**
Требования описывают механизмы обнаружения контента:
*   **REQ-SEARCH-002:** Полнотекстовый поиск по содержимому и метаданным через `ElasticsearchCluster`.
*   **REQ-SEARCH-003:** Поддержка серверной пагинации результатов (`page`, `pageSize`).
*   **REQ-CATALOG-001/002:** Хранение метаданных в MongoDB и их обогащение через внешнюю базу ISBN.

**4.1.3. Управление контентом и офлайн-режим (REQ-CONTENT, REQ-REL)**
Обеспечение доступности книг:
*   **REQ-CONTENT-002:** Реализация паттерна кэширования (Redis) перед обращением к объектному хранилищу.
*   **REQ-CONTENT-004/005:** Сохранение загруженного контента в `OfflineStorage` и его чтение `BookViewer` при отсутствии сети (обеспечение требования надежности **REQ-REL-001**).
*   **REQ-CONTENT-003:** Проверка целостности бинарных данных через контрольную сумму (`checksum`).

**4.1.4. Синхронизация и пользовательский опыт (REQ-SYNC, REQ-BOOKMARK-PROGRESS)**
Требования к непрерывности работы на разных устройствах:
*   **REQ-SYNC-001/002:** Двусторонняя синхронизация изменений между `LocalCache` и сервером.
*   **REQ-PROGRESS-001:** Отслеживание позиции чтения и отправка событий в аналитику.

**4.1.5. Бизнес-логика: Бронирование и Профиль (REQ-BOOKING, REQ-USER, REQ-PROFILE)**
Правила выдачи литературы:
*   **REQ-BOOKING-001:** Фиксация факта бронирования в реляционной БД (`PostgreSQLCluster`).
*   **REQ-PROFILE-002:** Валидация лимита книг (проверка `borrowedBooksCount` <= `maxBorrowLimit`).
*   **REQ-LMS-001:** Импорт данных о курсах студента из LMS для персонализации.

**4.1.6. Инфраструктура: Уведомления, Аналитика, Мониторинг (REQ-NOTIFY, REQ-ANALYTICS, REQ-MONITOR)**
*   **REQ-NOTIFY-001:** Асинхронная обработка уведомлений через очереди RabbitMQ.
*   **REQ-ANALYTICS-002:** Потоковая передача событий в Kafka.
*   **REQ-MONITOR-001:** Сбор технических метрик (`cpuUsage`, `requestCount`) через интерфейс `ISystemMetrics`.

#### 4.2. Нефункциональные требования (Quality Attributes)

Отдельно выделена группа требований **REQ-PERF**, определяющая метрики производительности (SLO):
1.  **REQ-PERF-001:** Время ответа поиска не должно превышать **500 мс** для 95-го перцентиля запросов.
2.  **REQ-PERF-002:** Время доступа к кэшированным данным (Redis) — менее **10 мс**.

#### 4.3. Привязка требований к архитектурным компонентам (Binding)

В модели установлены связи между требованиями и реализующими их компонентами. Ниже приведена матрица трассируемости для ключевых узлов:

| Компонент                   | Реализуемые требования           | Обоснование привязки                                                       |
| :-------------------------- | :------------------------------- | :------------------------------------------------------------------------- |
| **AuthenticationService**   | REQ-AUTH-001, REQ-AUTH-002       | Сервис отвечает за взаимодействие с LDAP и эмиссию токенов.                |
| **SearchService**           | REQ-SEARCH-001, REQ-PERF-001     | Обеспечивает интерфейс поиска и гарантирует время ответа < 500мс.          |
| **ContentDeliveryService**  | REQ-CONTENT-002, REQ-PERF-002    | Реализует логику кэширования в Redis для ускорения отдачи контента.        |
| **SyncEngine (Client)**     | REQ-SYNC-001, REQ-REL-002        | Управляет очередью локальных изменений и повторяет попытки при сбоях сети. |
| **BookingService**          | REQ-BOOKING-001, REQ-PROFILE-002 | Содержит логику проверки лимитов выдачи и записи транзакций.               |
| **OfflineStorage (Client)** | REQ-CONTENT-004, REQ-REL-001     | Физически хранит файлы для работы без интернета.                           |
| **HealthMonitor**           | REQ-MONITOR-001, REQ-MONITOR-003 | Агрегирует системные метрики и передает их инфраструктуре мониторинга.     |
| **AnalyticsService**        | REQ-ANALYTICS-002                | Выступает продьюсером (Producer) для кластера Kafka.                       |

#### 4.4. Верификация требований через интерфейсы

Формализация требований также отражена в проектировании интерфейсов данных:
*   Требование **REQ-AUTH-003** (время жизни токена) реализовано полем `expiresIn` в интерфейсе `IAuthToken`.
*   Требование **REQ-SEARCH-004** (возврат метаданных) реализовано включением массива `IBookMetadata` в структуру `ISearchResults`.
*   Требование **REQ-SEC-001** (безопасность) диктует обработку поля `password` интерфейса `IUserCredentials`.

Таким образом, функциональные требования не просто декларированы, а "зашиты" в структуру данных и распределение ответственности между компонентами системы.

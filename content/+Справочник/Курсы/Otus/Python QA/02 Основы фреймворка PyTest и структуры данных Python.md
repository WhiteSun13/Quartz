## 🤔 Разминка — что такое тестовый фреймворк?

> [!important] Определение
> **Тестовый фреймворк** — это совокупность методик, подходов и инструментов для написания тестов. Он задаёт вам «колею» — говорит: «Вот есть какие-то штуки, используй их вот так, и всё будет работать».

### Фреймворк vs Библиотека

| Фреймворк | Библиотека |
|-----------|-----------|
| Говорит: «Пиши **здесь**, положи **то-то** в этот файлик, объяви **вот так**» | Говорит: «Вот есть функция, она работает так, используй её **где хочешь**» |
| Задаёт **структуру** и правила | Предоставляет **инструменты** без навязывания структуры |
| Управляет потоком выполнения | Вы управляете потоком выполнения |

### Можно ли писать автотесты без тестового фреймворка?

> [!tip] Ответ
> **Да, можно.** Можно писать функции, добавлять `assert`, обрабатывать результаты. Но в итоге вы придёте к тому, что напишете **свой собственный фреймворк**. Именно так фреймворки и появились — кто-то тестировал свой код, получился фреймворк, выложил в открытый доступ.

## ⚡ Почему PyTest?

### Преимущества PyTest

| Преимущество | Описание |
|-------------|----------|
| **Многоцелевой** | Юнит-тестирование, API-тестирование, интерфейсное тестирование |
| **Плагины** | Возможность писать и устанавливать плагины (например, распараллеливание тестов, красивый вывод, Allure-отчёты) |
| **Простой в использовании** | Стандартный `assert` Python вместо `self.assertEqual` из unittest |
| **Отличная документация** | Подробная, с хорошим комьюнити |
| **Распространённость** | Очень распространённый фреймворк, много решений в сети |

#### Сравнение с unittest:

```python
# unittest
self.assertEqual(a, b)

# PyTest — стандартный assert Python
assert a == b
```

> [!tip] Рекомендация
> Сохраните себе в закладки [документацию PyTest](https://docs.pytest.org/) и почаще в ней копайтесь — там очень много интересного и подробно описанного.

### Недостаток PyTest

> [!warning] Внешняя зависимость
> PyTest **не является частью стандартной библиотеки Python** (в отличие от unittest). Это накладывает обязанности:
> - Следить за **версией**
> - Решать вопросы **совместимости с плагинами**
> - Возможны **баги** в новых версиях

#### Практический опыт с обновлениями:

> [!quote] Из опыта лектора
> «В последних версиях были несколько неприятных вещей, которые нас сильно аффектили, и нам пришлось откатываться назад на предыдущую версию. Мы закладываем, что раз в полтора-два месяца нужно проверять обновления. Не бросаемся на свежий релиз — сначала чекаем, насколько отстали, смотрим release notes, обновляемся на максимально стабильную версию, ловим баги, степаем назад до устраивающей версии.»

> [!danger] Важное правило
> **На свежий релиз никогда не обновляйтесь сразу!** Тщательно проверьте, на что обновились. Не запускайте процесс обновления — иначе потом будет тяжелее.

> [!important] Несмотря на всё это
> PyTest на сегодня в Python — **must have**. Если вы говорите, что вы тестировщик-автоматизатор на Python, вы **должны** знать PyTest.

### 🔍 Что такое виртуальное окружение?

> [!important] Определение
> **Виртуальное окружение (virtual environment)** — это изоляция Python внутри вашего компьютера. Зависимости устанавливаются **в проект**, а не в глобальную систему.

#### Зачем это нужно?

- У вас может быть **соседний проект** с PyTest версии 4.6, а в текущем — другая версия
- Зависимости **не конфликтуют** между проектами
- Вы не тащите зависимости в свой компьютер глобально

#### Проверка активации:

```bash
which python    # Должен показать путь внутри папки venv
which pip       # Аналогично
```

> [!warning] Для macOS / Linux
> Команда `python` может ссылаться на Python 2 (особенно на macOS). Используйте **`python3`**. **Не меняйте** системную ссылку `python` — это может сломать систему (особенно Linux).

> [!note] Модуль venv
> `venv` — стандартный модуль Python. Есть и другие обёртки (`virtualenv`, `pyenv`), но `venv` — базовый и достаточный.

## 📂 Основы PyTest — соглашения об именовании

### Правила именования

Чтобы PyTest автоматически находил тесты:

| Элемент | Правило именования |
|---------|-------------------|
| **Файлы** | Префикс `test_` (например, `test_login.py`) |
| **Классы** | Префикс `Test` (например, `TestLogin`) |
| **Функции/методы** | Префикс `test_` (например, `test_valid_login`) |

> [!warning] Правило хорошего тона
> Всегда ставьте **подчёркивание** после слова `test`: `test_something`, а не `testsomething`. Иначе ревьюеры и разработчики вас поправят.

> [!note] Переопределение
> В PyTest можно переопределить **всё** — включая префиксы файлов, классов и функций, а также имя файла `conftest`. Но **не рекомендуется** это делать — если что-то пойдёт не так, никто, кроме вас, не сможет помочь.

## 🚀 Запуск тестов — флаги и параметры

### Основные команды запуска

```bash
# Запустить все тесты в проекте
pytest

# Запустить тесты в конкретной папке
pytest 0_pytest_intro/

# Запустить конкретный файл (модуль)
pytest 0_pytest_intro/test_example.py

# Запустить конкретную тестовую функцию
pytest 0_pytest_intro/test_example.py::test_1
```

### Флаги управления выводом

| Флаг | Описание |
|------|----------|
| `-v` | **Verbose** — подробный вывод (название каждой функции, статус) |
| `-q` | **Quiet** — минималистичный вывод (только точки и итог) |
| `-s` | Показать **print-вывод** (по умолчанию PyTest перехватывает stdout) |

```bash
# Подробный вывод
pytest -v

# Минимальный вывод
pytest -q

# Показать принты
pytest -s

# Комбинация: подробный + принты
pytest -s -v
```

### Флаги управления запуском

| Флаг | Описание |
|------|----------|
| `-x` | Остановиться после **первого** падения |
| `--maxfail=N` | Остановиться после **N** падений |
| `--collect-only` | Только **собрать** информацию о тестах (не запускать) |
| `--lf` | Запустить только **последние упавшие** тесты (Last Failed) |
| `-h` | **Help** — показать все доступные параметры |

> [!tip] Когда использовать `-x`
> В юнит-тестах: если тест не прошёл — значит, код сломан. Разработчику не нужно ждать 20 минут прогона всех тестов, чтобы узнать о проблеме.

> [!tip] Когда использовать `--lf`
> Утром приходите, видите 5 упавших тестов из ночного запуска. Подозрение, что тестовая среда «моргнула». Перезапускаете только упавшие — если прошли, значит, было моргание среды.

## 🔧 Фикстуры (Fixtures) — киллер-фича PyTest

### 📌 Что такое фикстура?

> [!important] Определение
> **Фикстура** — это особая функция, помеченная декоратором `@pytest.fixture`, которая вызывается PyTest-ом перед (и/или после) тестовых функций. Это аналог **setup/teardown** из других фреймворков.

```python
import pytest

@pytest.fixture
def first_fixture():
    print("Print from first fixture")

def test_1(first_fixture):
    pass

def test_2(first_fixture):
    pass
```

> [!note] Как это работает
> 1. PyTest видит, что в тестовую функцию передан параметр `first_fixture`
> 2. Проверяет — это не стандартное имя, значит, это фикстура
> 3. Ищет функцию с декоратором `@pytest.fixture` и таким именем
> 4. Выполняет код фикстуры перед запуском теста

#### Пример использования:

Вместо того чтобы в каждом тесте писать подключение к базе данных, вы выносите это в фикстуру:

```python
@pytest.fixture
def db_connection():
    conn = connect_to_db(host="...", user="admin", password="...")
    return conn
```

### Фикстура vs обычная функция

> [!warning] Важное отличие
> Фикстура — это **особая сущность** для PyTest. Без декоратора `@pytest.fixture` PyTest не сможет найти и использовать функцию как фикстуру. Обычная функция с тем же именем **не будет работать** как фикстура.

```python
# Это фикстура — PyTest её найдёт
@pytest.fixture
def my_fixture():
    print("I'm a fixture")

# Это обычная функция — PyTest её НЕ найдёт как фикстуру
def my_fixture():
    print("I'm NOT a fixture")
```

### 📁 Поиск фикстур и файл conftest.py

#### Порядок поиска фикстур (снизу вверх):

```
1. В самом тестовом файле
2. В conftest.py рядом с тестовым файлом
3. В conftest.py на уровень выше
4. В conftest.py в корне проекта
```

> [!important] Принцип «лестницы»
> PyTest ищет фикстуру **снизу вверх** — от текущего файла к корню проекта. Если не находит нигде — выдаёт ошибку.

#### Пример структуры:

```
project/
├── conftest.py              # 4. Глобальный conftest (корень)
├── tests/
│   ├── conftest.py          # 3. conftest уровня tests/
│   ├── module_a/
│   │   ├── conftest.py      # 2. conftest рядом с тестом
│   │   └── test_example.py  # 1. Сам тестовый файл
```

> [!tip] Зачем использовать conftest.py
> Если разные тестовые файлы используют **одну и ту же фикстуру** — вынесите её в `conftest.py`. Тогда она будет доступна всем тестам в этой директории и ниже.

### 🎯 Scope фикстур (область действия)

Фикстуры могут работать на разных **уровнях**:

| Scope | Описание | Когда создаётся / уничтожается |
|-------|----------|-------------------------------|
| `function` | По умолчанию | Для **каждой** тестовой функции |
| `class` | Уровень класса | Один раз для **всего тестового класса** |
| `module` | Уровень модуля (файла) | Один раз для **всего файла** |
| `session` | Уровень сессии | Один раз для **всего запуска тестов** |
| `package` | Экспериментальный | Для пакета (появился недавно) |

```python
@pytest.fixture(scope="function")
def function_fixture():
    print("Hello from function fixture")

@pytest.fixture(scope="class")
def class_fixture():
    print("Hello from class fixture")

@pytest.fixture(scope="module")
def module_fixture():
    print("Hello from module fixture")

@pytest.fixture(scope="session")
def session_fixture():
    print("Hello from session fixture")
```

> [!important] Принцип «матрёшки»
> Фикстуры складываются как **матрёшка**:
> - **Сессия** открывается первой, закрывается последней
> - **Модуль** внутри сессии
> - **Класс** внутри модуля
> - **Функция** внутри класса
>
> Открытие: session → module → class → function
> Закрытие: function → class → module → session

### ⚠️ Auto-use — автоматическое использование фикстур

```python
@pytest.fixture(autouse=True)
def always_used_fixture():
    print("I run for every test automatically!")
```

Фикстура с `autouse=True` вызывается **автоматически** для каждого теста на своём уровне scope, **без передачи** в параметры тестовой функции.

> [!danger] Опасность auto-use
> **Auto-use — коварная и опасная штука!**
> - Когда проект растёт и разработчиков становится много, другой разработчик **не ожидает** такого поведения
> - Фикстура может «лезть» туда, где вы не ожидали
> - **Плохая практика** — если есть сомнения, лучше не использовать
> - Всегда старайтесь **явно передавать** фикстуры

> [!tip] Совет
> Если вы на 100% уверены, что auto-use нужен, и уверены, что **все в команде поймут** — используйте. В остальных случаях — **не трогайте этот параметр**.

### 📦 Объект request

> [!info] Что такое request?
> Каждая фикстура **неявно** принимает объект `request`. Он содержит много служебной информации о текущем запуске.

```python
@pytest.fixture
def my_fixture(request):
    print(request.node)      # Откуда вызывается (нода)
    print(request.scope)     # Scope фикстуры
    print(request.cls)       # Класс (или None)
    print(request.fspath)    # Полный путь до модуля
```

#### Содержимое объекта request:

Объект содержит очень много информации — ключевые слова, instance, пути, scope и другое. Многие **плагины PyTest** строятся на его основе.

### 🔚 Финализация фикстур (Teardown)

Два способа реализовать **финализацию** (очистку после тестов):

#### Способ 1: через `request.addfinalizer`

```python
@pytest.fixture
def db_fixture(request):
    # Setup
    conn = connect_to_db()
    print("Connected to DB")
    
    def fin():
        # Teardown
        conn.close()
        print("Connection closed")
    
    request.addfinalizer(fin)
    return conn
```

#### Способ 2: через `yield`

```python
@pytest.fixture
def db_fixture():
    # Setup (до yield)
    conn = connect_to_db()
    print("Connected to DB")
    
    yield conn  # Возвращает значение в тест
    
    # Teardown (после yield)
    conn.close()
    print("Connection closed")
```

> [!tip] Рекомендация лектора
> Через `addfinalizer` делать **прозрачнее и понятнее**. Некоторые разработчики, использующие PyTest, вообще не знают про способ с `yield`. Но если увидите `yield` в фикстуре — знайте: **до yield** — setup, **после yield** — teardown.

### 🪆 Вложенность фикстур (фикстура в фикстуре)

Фикстура может использовать **другую фикстуру**:

```python
@pytest.fixture
def fixture_1():
    print("Hi from fixture 1")
    yield
    print("Bye from fixture 1")

@pytest.fixture
def fixture_2(fixture_1):  # Принимает fixture_1
    print("Hi from fixture 2")
    yield
    print("Bye from fixture 2")

def test_example(fixture_2):  # Передаём только fixture_2
    print("Test body")
```

#### Порядок выполнения:

```
Hi from fixture 1     ← fixture_1 setup
Hi from fixture 2     ← fixture_2 setup
Test body             ← тест
Bye from fixture 2    ← fixture_2 teardown
Bye from fixture 1    ← fixture_1 teardown
```

> [!important] Принцип работы
> PyTest ищет **самую глубокую** фикстуру, которая ни от чего не зависит, и начинает выполнение оттуда. Это похоже на **рекурсию** — вызовы идут вглубь, пока не найдётся «дно» (фикстура без параметров), и затем выполнение разворачивается обратно.

> [!tip] Практическое применение
> Одна фикстура делает **соединение с БД**, другая на основе этого соединения **генерирует тестовые данные**, а потом данные удаляются, но соединение не закрывается. Матрёшку можно складывать на любую глубину.

### 🔄 Возвращаемые значения из фикстур

Фикстура может **возвращать значение** через `return` — и это значение будет доступно в тесте:

#### Пример: возврат простого значения

```python
@pytest.fixture
def random_number():
    return random.randint(1, 100)

def test_check_number(random_number):
    # random_number теперь содержит случайное число
    assert random_number == 42  # Будет падать, т.к. число случайное
```

> [!important] Ключевой принцип
> Когда фикстура делает `return`, она **превращается** в то, что вернула. В тесте вместо имени фикстуры — **возвращённое значение**.

#### Пример: возврат экземпляра класса

```python
class TestHelper:
    def __init__(self, mod1, mod2):
        self.mod1 = mod1
        self.mod2 = mod2
    
    def hello(self, name):
        return f"Hello, {name}"

@pytest.fixture
def helper():
    return TestHelper(mod1=random, mod2=math)

def test_hello(helper):
    assert helper.hello("Вася") == "Hello, Вася"

def test_math(helper):
    assert helper.mod2.pow(2, 3) == 8

def test_random(helper):
    result = helper.mod1.choice(["a", "b", "c"])
    assert result == "a"  # Может упасть — значение случайное
```

> [!tip] Применение
> Возвращать из фикстуры можно:
> - Простое **значение** (число, строка)
> - **Объект** соединения к базе данных
> - **Экземпляр класса** с нужными методами
> - **Модуль** Python
> - Случайного **пользователя** из БД

### 📊 Параметризация

#### Параметризация тестовой функции:

```python
@pytest.mark.parametrize("test_input", [1, 2, 3])
def test_example(test_input):
    print(test_input)
```

> [!warning] Правило
> Имя параметра в `@pytest.mark.parametrize("test_input", ...)` должно **совпадать** с именем аргумента функции `def test_example(test_input)`.

#### Параметризация тестового класса:

```python
@pytest.mark.parametrize("test_input", [1, 2, 3])
class TestExample:
    def test_one(self, test_input):
        print(test_input)
    
    def test_two(self, test_input):
        print(test_input)
```

Результат: **3 значения × 2 теста = 6 запусков**

#### Параметризация фикстуры:

```python
@pytest.fixture(params=[1, 2, 3])
def parametrized_fixture(request):
    return request.param
```

#### Комбинаторика параметров:

Когда параметризованы **и фикстура, и тестовая функция** — каждый параметр фикстуры комбинируется с каждым параметром функции.

> [!info] Где особенно полезна параметризация
> - **API-тестирование** — проверка разных входных значений (минус один, NaN, бесконечность, пустая строка)
> - Один тест, одно место — **множество значений**
> - В UI-тестировании используется реже (тесты дорогие по времени)

## 🐍 Структуры данных Python

### Обзор основных структур

| Структура | Тип | Упорядоченная | Изменяемая | Допускает дубликаты |
|-----------|-----|:---:|:---:|:---:|
| **list** (список) | `[]` | ✅ | ✅ | ✅ |
| **tuple** (кортеж) | `()` | ✅ | ❌ | ✅ |
| **set** (множество) | `{}` | ❌ | ✅ | ❌ |
| **dict** (словарь) | `{k:v}` | ❌* | ✅ | ❌ (ключи) |

*В больших словарях порядок не гарантирован

---

### 📋 Список (list)

#### Создание и основные операции (CRUD):

```python
# Создание
l = []
l = [1, 2, 3]

# Добавление в конец
l.append(1)

# Присоединение другого списка
l.extend([2, 3, 4])

# Вставка по индексу
l.insert(0, 1)      # В позицию 0 вставить значение 1

# Замена значения по индексу
l[1] = 103

# Удаление элемента по индексу
del l[0]

# Подсчёт вхождений
l.count(3)

# Проверка вхождения
1 in l  # True или False

# Удаление всего списка
del l
```

#### List Comprehension (генерация списков):

```python
# Чётные числа от 0 до 50
l = [x for x in range(50) if x % 2 == 0]
```

> [!tip] List Comprehension
> Очень полезная конструкция — позволяет создавать списки с циклами и условиями в одну строку. Особенно эффективна при генерации больших списков.

#### Слайсы (нарезка):

```python
l = [x for x in range(50)]

l[5:30:2]    # От 5 до 30 с шагом 2
l[:30]       # От начала до 30
l[::2]       # Все с шагом 2
l[::-1]      # Перевернуть список
```

**Формат:** `список[начало:конец:шаг]`

#### Итерация:

```python
for element in l:
    print(element)
```

> [!note] Строка как список
> Строка — это своеобразный список символов. По ней тоже можно итерироваться, слайсить, доставать символы по индексу.

#### ⚠️ Копирование списков:

> [!danger] Опасность передачи по ссылке
> ```python
> l1 = [1, 2, 3]
> l3 = l1          # ЭТО НЕ КОПИЯ! Это ссылка на тот же объект
> l3.extend([4, 5])
> print(l1)        # [1, 2, 3, 4, 5] — l1 тоже изменился!
> ```

**Правильные способы копирования:**

```python
# Способ 1: метод copy()
l2 = l1.copy()

# Способ 2: пустой слайс
l4 = l1[:]
```

#### Реверс списка:

```python
l = [1, 2, 3, 4, 5]

# Способ 1: l.reverse() — изменяет ИСХОДНЫЙ список, возвращает None
l.reverse()     # l теперь [5, 4, 3, 2, 1]

# Способ 2: reversed() — возвращает итератор, НЕ меняет исходный
lr = list(reversed(l))

# Способ 3: слайс с шагом -1 — НЕ меняет исходный, сразу список
lr = l[::-1]
```

| Способ | Меняет исходный | Возвращает |
|--------|:---:|------------|
| `l.reverse()` | ✅ | `None` |
| `reversed(l)` | ❌ | Итератор |
| `l[::-1]` | ❌ | Новый список |

### 📦 Кортеж (tuple)

```python
t = (1, "hello", 3.14)

# Доступ по индексу
t[0]        # 1

# Нельзя изменять!
t[1] = "world"  # TypeError: 'tuple' does not support item assignment

# Итерация
for item in t:
    print(item)

# Длина
len(t)      # 3

# Переопределение (создание нового кортежа)
t = (4, 5, 6)  # Можно — это новый объект
```

> [!important] Ключевая особенность
> **Кортеж неизменяемый.** Нельзя изменить отдельный элемент — только создать новый кортеж целиком. Это **не константа** — это способ хранения данных.

**Когда использовать кортеж:**
- **Координаты** (широта, долгота) — неразделимые значения
- **Точки маршрута** — фиксированные данные
- Как **ключ в словаре** (список ключом быть не может!)

> [!tip] Совет
> Если не знаете, зачем вам прямо сейчас кортеж — используйте **список**. На большом масштабе кортеж экономит ресурсы (не хранит индексы, не пересчитывает).

### 🎯 Множество (set)

```python
# Создание
s = set()
s.add(1)
s.add(2)
s.add(1)    # Дубликат — не добавится
print(s)    # {1, 2}

# Второй способ
s = {1, 2, 3, 4, 3, 4, 5, 6}
print(s)    # {1, 2, 3, 4, 5, 6} — без дубликатов
```

#### Операции над множествами:

```python
s1 = {1, 2, 3, 4}
s2 = {3, 4, 5, 6}

# Пересечение
s1.intersection(s2)    # {3, 4}

# Разность
s1.difference(s2)      # {1, 2}

# Объединение
s1.union(s2)           # {1, 2, 3, 4, 5, 6}
```

> [!note] Особенности set
> - **Без повторений** — автоматически удаляет дубликаты
> - **Неупорядоченный** — нельзя обратиться по индексу
> - `s.pop()` — достаёт **случайный** элемент (не гарантирован порядок)

### 📖 Словарь (dict)

```python
# Создание
d = {}
d = {"key": "value", "name": "John"}

# Добавление / изменение
d["age"] = 25           # Добавить
d["name"] = "Jane"      # Переопределить

# Получение значения
d["name"]               # "Jane"

# Удаление
del d["age"]            # Удалить пару ключ-значение
del d                   # Удалить весь словарь
```

#### Итерация по словарю:

```python
d = {"a": 1, "b": 2, "c": 3}

# По ключам (по умолчанию)
for key in d:
    print(key)          # a, b, c

# По значениям
for value in d.values():
    print(value)        # 1, 2, 3

# По парам ключ-значение
for key, value in d.items():
    print(f"{key}: {value}")
```

> [!warning] Порядок элементов
> На больших объёмах словарей **порядок не гарантирован**. Не рассчитывайте на определённый порядок при итерации.

> [!tip] OrderedDict
> Если нужен гарантированный порядок — используйте `collections.OrderedDict` из стандартной библиотеки. Также в `collections` есть другие полезные контейнеры — изучите их.

#### Связь с JSON:

> [!info] Словарь и JSON
> **JSON** (JavaScript Object Notation) и словарь Python очень тесно связаны. При тестировании **API** это будет основная структура данных для работы с запросами и ответами.

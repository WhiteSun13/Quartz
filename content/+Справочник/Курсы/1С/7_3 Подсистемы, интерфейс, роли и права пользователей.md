## 🏷 Настройка имени конфигурации

**Шаги:**

1. В дереве конфигурации нажать ПКМ на корне → **«Свойства конфигурации»**
2. **Имя** (на английском): **`Merchant`**

> [!note]
> Имя конфигурации задаётся на **английском языке**, т.к. 1С — база данных, и администратор работает с именами именно в английском виде. Это упрощает администрирование.

3. **Синоним** (отображается пользователям): **`1С Купец`**

> [!success]
> В пользовательском режиме сверху слева будет отображаться надпись **«1С Купец»**.

## 🗂 Подсистемы

### 💡 Что такое подсистемы

> [!important]
> **Подсистемы** — объекты конфигурации, предназначенные для **логической группировки** объектов конфигурации.
>
> Примеры:
> - Подсистема **«Бухгалтерия»** → все документы и справочники, связанные с бухгалтерией
> - Подсистема **«Торгово-складской учёт»** → всё, что связано с торговлей и складом

**По условию заказчика — 3 подсистемы:**

| Подсистема | Что входит |
|---|---|
| **Кадровый учёт** | Справочник «Сотрудники» |
| **НСИ** (нормативно-справочная информация) | Константы, форма «Данные организации» |
| **Торговый учёт** | Справочники, документы, перечисления, отчёты, регистры накопления |

---

### 🛠 Создание подсистем

**Расположение:** Дерево конфигурации → **«Общие»** → **«Подсистемы»**

---

#### 📁 Подсистема 1: Кадровый учёт

1. ПКМ → **«Добавить»**
2. Имя: **`КадровыйУчёт`**, синоним: **«Кадровый учёт»**
3. Вкладка **«Состав»** → раскрыть «Справочники» → поставить галочку: ☑ **Сотрудники**

> [!note]
> На профильном блоке эта подсистема будет дополняться: появятся документы, регистры и т.д.

---

#### 📁 Подсистема 2: НСИ

1. ПКМ → **«Добавить»**
2. Имя: **`NSI`**, синоним: **«НСИ»**
3. Вкладка **«Состав»**:
   - Раскрыть «Константы» → ☑ **НазваниеОрганизации**, ☑ **ИНН**, ☑ **Телефон**, ☑ **ГенеральныйДиректор**
   - Раскрыть **«Общие»** → «Общие формы» → ☑ **ДанныеОрганизации**

---

#### 📁 Подсистема 3: Торговый учёт

1. ПКМ → **«Добавить»**
2. Имя: **`ТорговыйУчёт`**, синоним: **«Торговый учёт»**
3. Вкладка **«Состав»**:

| Ветка | Объекты |
|---|---|
| Справочники | ☑ Номенклатура, ☑ Контрагенты, ☑ Договоры |
| Документы | ☑ ЗаказКлиента, ☑ ПриходнаяНакладная, ☑ РасходнаяНакладная |
| Перечисления | ☑ Все перечисления |
| Отчёты | ☑ ОтчётПоПродажам |
| Регистры накопления | ☑ Продажи |

---

### 🔍 Отбор объектов по подсистемам в дереве конфигурации

> [!tip]
> В дереве конфигурации можно отображать только объекты конкретной подсистемы:
> 1. Нажать кнопку **«Действия»** в дереве конфигурации
> 2. Выбрать **«По подсистемам»**
> 3. Указать нужную подсистему → нажать **«Установить»**
>
> Чтобы вернуть все объекты: **«Действия»** → **«По подсистемам»** → **«Отключить»**

## 🖥 Настройка интерфейса клиентского приложения

### ⚙️ Открытие редактора интерфейса

1. В дереве конфигурации ПКМ на **«Merchant»** (корень)
2. Выбрать **«Открыть интерфейс клиентского приложения»**

### 🔧 Настройка расположения панелей

| Действие | Что делаем |
|---|---|
| Убрать «Панель функций текущего раздела» | Выделить → нажать крестик ✕ |
| Переместить «Панель разделов» | Перетащить **влево** |
| Переместить «Панель открытых» | Переместить **вниз** |

![[Pasted image 20260517163409.png]]

> [!success]
> Итоговая структура: подсистемы отображаются **вертикально слева** — как в требованиях заказчика.

---

### 🖼 Настройка отображения панели разделов

1. В правой области выделить **«Панель разделов»** → двойной клик (открытие свойств)
2. Найти свойство **«Отображение»**

**Варианты отображения:**

| Вариант | Описание |
|---|---|
| Только картинка | Только иконка без текста |
| Картинка и текст | Иконка + название (по умолчанию) ✅ |
| Текст | Только текст |
| Текст справа от картинки | Горизонтальное расположение |
| Текст под картинкой | Вертикальное расположение |

→ Оставить: **«Картинка и текст»**

## 🎨 Общие картинки для подсистем

### 💡 Зачем нужны общие картинки

> [!note]
> По умолчанию платформа отображает жёлтые кружки возле названий подсистем. Мы заменим их на собственные картинки.

### 📥 Подготовка файлов

Скачать с Яндекс.Диска (папка «Картинки подсистем»):
- `ПодсистемаКадровыйУчёт2` — для подсистемы «Кадровый учёт»
- `ПодсистемаНСИ2` — для подсистемы «НСИ»
- `ПодсистемаТорговыйУчёт2` — для подсистемы «Торговый учёт»

---

### 🛠 Создание объектов «Общие картинки»

**Расположение:** Дерево конфигурации → **«Общие»** → **«Общие картинки»**

**Для каждой картинки:**

1. ПКМ → **«Добавить»**
2. Задать имя (например, **`КадровыйУчёт`**)
3. Нажать **«Выбрать из файла»**
4. Указать скачанный файл → **«Открыть»**
5. Картинка сохраняется автоматически → закрыть окно

**Создать три объекта:**

| Имя объекта | Файл картинки |
|---|---|
| `КадровыйУчёт` | ПодсистемаКадровыйУчёт2 |
| `ТорговыйУчёт` | ПодсистемаТорговыйУчёт2 |
| `НСИ` | ПодсистемаНСИ2 |

---

### 🔗 Привязка картинок к подсистемам

**Для каждой подсистемы:**

1. Раскрыть подсистему → вкладка **«Основное»** → прокрутить вниз
2. Найти свойство **«Картинка»** → нажать **«...»** (три точки)
3. Перейти на вкладку **«Из конфигурации»** → выбрать нужную картинку → **«Ок»**

> [!note]
> Также доступна вкладка **«Стандартные»** — там можно выбрать встроенные картинки платформы.

> [!tip]
> Чтобы изменить картинку **главного раздела** (лампочка слева от «Главная»):
> ПКМ на «Merchant» → **«Свойства»** → свойство **«Картинка основного раздела»** → нажать «Открыть» → выбрать картинку.

---

### 🧹 Убираем лишние константы из НСИ

> [!warning]
> После создания подсистем в НСИ снова появились отдельные константы (хотя мы это уже убирали в начале курса).

**Решение:**

1. Раскрыть подсистему **«НСИ»** → нажать **«Командный интерфейс»**
2. Снять флажки с констант:
   - ГенеральныйДиректор
   - ИНН
   - Телефон
   - ЮридическоеНаименование
3. Оставить только **«ДанныеОрганизации»** (форму)
4. Нажать **«Ок»**

## 📂 Группы команд

### 💡 Зачем нужны группы команд

> [!note]
> Группы команд позволяют **структурировать объекты внутри подсистемы** — создавать заголовки-разделы, группирующие команды в командном интерфейсе.

**Расположение:** Дерево конфигурации → **«Общие»** → **«Группы команд»**

### 🛠 Создание групп команд

1. ПКМ → **«Добавить»** → Имя: **`ТорговыеОперации`**, синоним: **«Торговые операции»**
2. ПКМ → **«Добавить»** → Имя: **`УчётныеДанные`**, синоним: **«Учётные данные»**

![[Pasted image 20260517191924.png]]

---

### ⚙️ Настройка командного интерфейса подсистем

#### Подсистема «Кадровый учёт»

1. Раскрыть → **«Командный интерфейс»**
2. Перетащить **«Сотрудники компании»** → в папку **«Учётные данные»**
3. Нажать **«Ок»**

![[Pasted image 20260517191946.png]]

#### Подсистема «Торговый учёт»

1. Раскрыть → **«Командный интерфейс»**
2. Перетащить в **«Учётные данные»**:
   - Договоры
   - Номенклатура
   - Список контрагентов
3. Перетащить в **«Торговые операции»**:
   - Заказы клиентов
   - Приходные накладные
   - Расходные накладные
4. Нажать **«Ок»**

**Итоговая структура «Торгового учёта»:**

```
Торговые операции
├── Заказы клиентов
├── Приходные накладные
└── Расходные накладные

Учётные данные
├── Договоры
├── Номенклатура
└── Список контрагентов
```

> [!success]
> После обновления конфигурации и запуска пользовательского режима интерфейс выглядит структурировано, как в требованиях заказчика.

## 🔐 Роли и права пользователей

### 🧩 Структура системы прав в 1С

```
Права (маленькие кружки)
    ↓ совокупность
Роли (средние кружки)
    ↓ назначаются
Пользователи (большие кружки)
```

![[Pasted image 20260517192840.png]]

> [!important]
> **Три уровня системы прав:**
>
> | Уровень | Описание |
> |---|---|
> | **Права** | Конкретные разрешения на объект (просмотр, запись, изменение, удаление) |
> | **Роли** | Набор прав, объект конфигурации |
> | **Пользователи** | Конкретные люди, которым назначаются роли |

> [!note]
> В типовых конфигурациях есть ещё уровень **«Профили доступа»** и **«Группы доступа»** — это дополнительный пользовательский уровень для назначения ролей. Это самописный механизм, не встроенный в платформу по умолчанию.

---

### ⚠️ Важный нюанс перед созданием пользователей

> [!warning]
> При попытке создать пользователя без предварительно настроенных ролей возникнет ошибка:
> *«После создания пользователя в системе не осталось бы ни одного пользователя с административными правами»*
>
> **Решение:** Сначала создать роли, потом — пользователей.

---

### 🛠 Создание ролей

**Расположение:** Дерево конфигурации → **«Общие»** → **«Роли»**

---

#### 🔑 Роль 1: Администрирование

1. ПКМ → **«Добавить»**
2. Имя: **`Администрирование`**
3. В окне редактирования роли: **«Действие»** → **«Снять все права»**
4. Найти и поставить только две галочки:
   - ☑ **Администрирование**
   - ☑ **Администрирование данных** (ставится автоматически вместе с первым)

**Дополнительные флажки внизу окна:**

| Флажок | Назначение | Установить? |
|---|---|---|
| **Устанавливать права для новых объектов** | При создании новых объектов права добавляются автоматически | ☑ Да |
| **Установить права для реквизитов и табличных частей по умолчанию** | Права автоматически распространяются на реквизиты и табличные части | — |
| **Независимые права подчинённых объектов** | Позволяет выдать права на реквизит без прав на весь объект | — |

![[Pasted image 20260517193240.png]]

---

#### 🔑 Роль 2: Работа с данными

1. ПКМ → **«Добавить»**
2. Имя: **`РаботаСДанными`**, синоним: **«Работа с данными»**
3. В окне редактирования: **«Действие»** → **«Установить все права»**
4. Выделить объект **«Merchant»** (конфигурацию) → снять две галочки:
   - ☐ **Администрирование**
   - ☐ **Администрирование данных**
5. Установить флажок: ☑ **Устанавливать права для новых объектов**

> [!important]
> **Стандарт:** По рекомендациям права в разных ролях **не должны пересекаться**. Технически пересечение не критично, но по стандартам нежелательно.

> [!note]
> **Почему не устанавливать права вручную галочками?**
> Кнопки ☑/☐ в интерфейсе работают только для **конкретного выделенного объекта**. Используйте **«Действие» → «Установить все права»** — это применит права ко всем объектам конфигурации сразу.

---

### 👤 Создание пользователей

**Расположение:** Конфигуратор → вкладка **«Администрирование»** → **«Пользователи»**

> [!warning]
> Перед созданием пользователей нужно **обновить конфигурацию** (нажать синий бочонок), чтобы созданные роли появились в списке.

---

#### Пользователь 1: Администратор

1. Нажать **«+»** (добавить)
2. Имя: **`Администратор`**
3. Вкладка **«Прочие»** → поставить галочки:
   - ☑ **Администрирование**
   - ☑ **Работа с данными**
4. Нажать **«Ок»**

---

#### Пользователь 2: Пользователь

1. Нажать **«+»** (добавить)
2. Имя: **`Пользователь`**
3. Вкладка **«Прочие»** → поставить галочку:
   - ☑ **Работа с данными**
4. Нажать **«Ок»**

> [!note]
> Если у пользователя не установлено ни одной роли, в списке рядом с ним отображается **знак вопроса «?»**.

---

### 🔒 Работа с паролями

> [!warning]
> **Ограничение учебной версии платформы:** Установить пароль пользователю **невозможно**. Это сделано специально, чтобы начинающий разработчик не потерял доступ к базе.

> [!info]
> **Политика паролей (для коммерческих версий):**
> Настраивается через: **Администрирование** → **«Параметры информационной базы»** → раздел **«Политика паролей»**
>
> Доступные настройки:
> - Проверка сложности пароля
> - Минимальная длина пароля
> - Минимальный срок действия (в секундах)
> - Максимальный срок действия
>
> Если значения = 0 → ограничений нет.

---

### 🆚 Разница между администратором и пользователем

> [!important]
> **Тест: попытка открыть конфигуратор под пользователем (без прав администрирования)**
>
> → Выйдет предупреждение: **«Недостаточно прав для доступа»**
>
> Пользователь **не может:**
> - Открыть дерево конфигурации
> - Изменять конфигурацию
> - Создавать новые объекты
>
> Пользователь **может:**
> - Свободно работать со всеми объектами в **пользовательском режиме**

---

### ❓ Ответы на вопросы по правам и пользователям

> [!note]
> **«Как поменять пароль, если забыл?»**
> - Администратор может сбросить/поменять пароль другому пользователю
> - Можно создать обработку, где пользователь сам меняет свой пароль (администратор не будет знать нового пароля)
> - Использовать резервную копию базы без пароля — крайний вариант, часть данных будет потеряна

> [!note]
> **«Можно ли задать права на время (например, на неделю)?»**
> Стандартными средствами (кнопками) — нет. Только **программно** через написание специальной обработки.

> [!note]
> **«Что такое РЛС?»**
> **Ограничение на уровне записей (РЛС)** — механизм, позволяющий скрывать от пользователя часть записей объекта, не лишая его прав на объект в целом. Тема для продвинутого уровня.

> [!note]
> **«Почему не рассматривался командный интерфейс старых версий?»**
> Современные конфигурации используют интерфейс **«Такси»**. Старые варианты интерфейса (похожие на конфигуратор с вкладками сверху) уже не актуальны и не рассматриваются в курсе.

### 🔑 Ключевые понятия занятия

| Понятие | Описание |
|---|---|
| **Подсистема** | Логическая группа объектов конфигурации; отображается как раздел в меню |
| **Командный интерфейс** | Настройка видимости и расположения команд внутри подсистемы |
| **Группа команд** | Объект для создания заголовков-папок внутри командного интерфейса |
| **Общие картинки** | Объект конфигурации для хранения и использования изображений |
| **Роль** | Объект конфигурации — набор прав для последующего назначения пользователям |
| **Права** | Конкретные разрешения на операции с объектом (просмотр, запись, удаление) |
| **Пользователь** | Учётная запись с назначенными ролями |
| **Администрирование** | Право, дающее доступ к конфигуратору и изменению конфигурации |
| **РЛС** | Ограничение на уровне записей — скрытие части данных от пользователя |
| **Политика паролей** | Настройки сложности, длины и срока действия паролей (Администрирование → Параметры ИБ) |
| **Интерфейс «Такси»** | Современный интерфейс 1С, используемый в актуальных конфигурациях |

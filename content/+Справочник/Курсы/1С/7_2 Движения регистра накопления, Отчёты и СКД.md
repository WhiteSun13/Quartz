# 📚 Конспект занятия: Движения регистра и Отчёты в 1С

---

## 🔄 Настройка алгоритма движений в регистр накопления

### 💡 Контекст

> [!important]
> До перерыва была настроена **связь** документа «Расходная накладная» с регистром накопления «Продажи» (назначен регистратор). Однако **алгоритм передачи данных** из документа в регистр задан не был — регистр оставался пустым.

---

### 🛠 Вкладка «Движения» в документе

**Шаги:**

1. В дереве конфигурации раскрыть ветку **«Документы»**
2. Двойной клик по **«Расходная накладная»**
3. Перейти на вкладку **«Движения»**

> [!note]
> Перенос данных из документа в регистры называется **«Движения»**. На этой вкладке можно увидеть, что регистр **«Продажи»** уже связан с документом.

---

### 🏗 Конструктор движений

> [!tip]
> Вместо ручного написания кода используем **«Конструктор движений»** — специальный инструмент, который генерирует код автоматически.

**Запуск:** На вкладке «Движения» нажать кнопку **«Конструктор движений»**.

#### Признак вида регистра в конструкторе

> [!info]
> **Как определить вид регистра по иконке:**
> 
> | Иконка | Вид регистра |
> |---|---|
> | 🟡 Жёлтая точка | **Обороты** |
> | ➕ Плюсик | Остатки (приход) |
> | ➖ Минусик | Остатки (расход) |

---

### 📋 Структура окна конструктора движений

| Область | Описание |
|---|---|
| **Верхняя левая** | Регистры — куда записываем данные (в нашем случае: «Продажи») |
| **Верхняя правая** | Реквизиты документа (дата, номер и все созданные реквизиты) |
| **Настройка «Таблица»** | Выбор табличной части, из которой берём данные |

---

### ⚙️ Заполнение конструктора — Табличная часть «Товары»

**Шаги:**

1. В настройке **«Таблица»** нажать стрелку вниз → выбрать **«Товары»**
2. В списке реквизитов дополнятся поля табличной части
3. Нажать кнопку **«Заполнить выражение»**

> [!note]
> **Результат автозаполнения:**
> - **Количество** → Количество (совпало имя и тип) ✅
> - **Сумма** → Сумма (совпало имя и тип) ✅
> - **Номенклатура** → ❌ (имена разные — нужно заполнить вручную)
> - **Контрагент** → ❌ (нужно заполнить вручную)

**Заполнение вручную:**

- **Номенклатура:** выделить поле → в списке реквизитов найти `ТекущаяСтрокаТовары.Товар` (галочка = совпадение типа) → выбрать
- **Контрагент:** доступен только **«Покупатель»** → выбрать его

---

### ➕ Добавление записи для Табличной части «Услуги»

> [!important]
> Данные нужно переносить из обеих табличных частей: **Товары** и **Услуги**.

**Шаги:**

1. Нажать **«+»** в верхней части конструктора (добавить новую запись)
2. Т.к. документ связан только с одним регистром — он подставится автоматически
3. Для второй записи выбрать **«Таблица: Услуги»**
4. Нажать **«Заполнить выражение»** (автоматически заполнится только «Сумма»)
5. Вручную:
   - **Номенклатура:** `ТекущаяСтрокаУслуги.Услуга`
   - **Контрагент:** `Покупатель`
   - **Количество:** вписать **`1`** (одна услуга за раз — доставка, ремонт и т.д.)

> [!tip]
> В поле **«Выражение»** можно указывать не только реквизиты, но и произвольные выражения: числа, формулы, обращения к данным алгоритма.

6. Нажать **«Ок»** → сформируется обработчик **«Обработка проведения»**

---

### 💻 Структура сгенерированного кода «Обработка проведения»

```
// Цикл по табличной части «Товары»
Для Каждого ТекущаяСтрокаТовары Из Товары Цикл
    Движение = Движения.Продажи.Добавить();
    Движение.Период      = Дата;
    Движение.Номенклатура = ТекущаяСтрокаТовары.Товар;
    Движение.Контрагент  = Покупатель;
    Движение.Количество  = ТекущаяСтрокаТовары.Количество;
    Движение.Сумма       = ТекущаяСтрокаТовары.Сумма;
КонецЦикла;

// Цикл по табличной части «Услуги»
Для Каждого ТекущаяСтрокаУслуги Из Услуги Цикл
    Движение = Движения.Продажи.Добавить();
    Движение.Период      = Дата;
    Движение.Номенклатура = ТекущаяСтрокаУслуги.Услуга;
    Движение.Контрагент  = Покупатель;
    Движение.Количество  = 1;  // фиксированное значение
    Движение.Сумма       = ТекущаяСтрокаУслуги.Сумма;
КонецЦикла;

Движения.Продажи.Записывать = Истина;
```

> [!important]
> **Свойство `Записывать = Истина`:**
> Это свойство определяет, будут ли записи реально помещены в регистр. Конструктор устанавливает его в `Истина` автоматически. Если значение `Ложь` — данные в регистр не попадут.

---

### 🚀 Проведение документов и проверка регистра

**Шаги:**

1. Обновить конфигурацию → запустить пользовательский режим
2. Перейти в **«Расходные накладные»**
3. Выделить все документы (зажать `Shift` → клик по первому и последнему)
4. ПКМ → **«Провести»**
5. Возле документов появятся галочки ✅

**Просмотр регистра:**

1. Нажать **«Сервис и настройки»** (верхний правый угол)
2. **«Функции для технического специалиста»**

> [!warning]
> Если пункт «Функции для технического специалиста» отсутствует:
> Перейти в **«Настройки» → «Параметры»** → поставить флажок **«Режим технического специалиста»**.

3. Раскрыть **«Регистры накопления»** → двойной клик по **«Продажи»**

> [!success]
> Теперь в регистре отображаются все записи, которые создали проведённые документы. Пример: регистратор 1, номенклатура «Видеокарта», контрагент «Белый Мишка», количество 10, сумма 150 000.

---

### 🔗 Вывод регистра в форму документа (удобный доступ)

> [!tip]
> Чтобы не заходить каждый раз в «Функции технического специалиста», можно вывести регистр прямо в форму расходной накладной.

**Шаги:**

1. Открыть любую расходную накладную
2. Нажать **«...» (три точки)** → пункт **«Окно»**
3. Выбрать **«Настройка панели навигации формы»**
4. В левой области раскрыть **«Перейти»**
5. Двойной клик по **«Продажи»** → они переместятся в правую область
6. Нажать **«Ок»**

> [!success]
> Сверху формы документа появится гиперссылка **«Продажи»**. Нажав на неё, увидим все движения по текущему документу.
>
> **Красные точки** слева от записей = регистр является оборотным.
> **Зелёные +/красные −** = регистр с остатками.

---

### ❓ Ответы на вопросы по движениям

> [!note]
> **«Если одних и тех же услуг несколько?»**
> Алгоритм запишет несколько записей (например, «Доставка», «Доставка»). Чтобы избежать дублей — нужно использовать **запрос** для схлопывания данных перед отправкой в регистр. Это тема более продвинутого уровня.

> [!note]
> **«Если продадим три принтера — появится одна запись?»**
> Да, появится одна запись с количеством 3 и соответствующей суммой.

## 📊 Объект конфигурации «Отчёты»

### 🧩 Назначение отчётов

> [!important]
> **Отчёты** — прикладной объект конфигурации, который используя **Систему компоновки данных (СКД)**, формирует таблицы или диаграммы для пользователя.

> [!quote]
> «Бухгалтер зайдёт в регистр накопления, посмотрит и ужаснётся. Поэтому мы сделаем красивый отчёт.»

**Три постулата архитектуры 1С:**

```
Документы → Регистры → Отчёты
```

---

### 🔑 Система компоновки данных (СКД)

```
┌─────────────────────────────────┐
│           СКД                   │
│  ┌──────────────────┐           │
│  │ Схема КД         │           │
│  │ (откуда данные,  │  ──────►  Отчёт на экране
│  │  как выводить)   │           │
│  └──────────────────┘           │
│  ┌──────────────────┐           │
│  │ Настройки        │           │
│  │ (внешний вид)    │           │
│  └──────────────────┘           │
└─────────────────────────────────┘
```

| Компонент | Описание |
|---|---|
| **Схема компоновки данных** | Инструкции: откуда брать данные, как их выводить, какие изменения применять |
| **Настройки** | Внешний вид отчёта, дополнительные параметры |

> [!note]
> Отчёты можно строить и без СКД, но тогда потребуется написать огромное количество кода вручную. СКД значительно упрощает этот процесс.

---

2. Выбрать тип: **«Запрос»**

#### Шаг 4. Конструктор запроса

> [!tip]
> Чтобы не писать запрос вручную — используем **«Конструктор запроса»** (аналог SQL, но на русском языке).

![[Pasted image 20260517143122.png]]

**Запуск:** Нажать кнопку **«Конструктор запроса»**

**Структура окна конструктора:**

| Область | Описание |
|---|---|
| **База данных** | Все таблицы и объекты базы данных |
| **Таблицы** | Таблицы, используемые в запросе |
| **Поля** | Поля, выбираемые для отчёта |

**Выбор таблицы:**

1. В области «База данных» раскрыть **«Регистры накопления»**
2. Найти две таблицы:
   - `Продажи` — физическая (реальная) таблица
   - `Продажи.Обороты` — **виртуальная таблица** (агрегированные данные)
3. Выбрать **`Продажи.Обороты`** (двойной клик или стрелка →)

> [!important]
> Используем виртуальную таблицу `Продажи.Обороты` — данные уже агрегированы, что быстрее и удобнее для отчёта.

**Выбор полей:**

1. Раскрыть таблицу `ПродажиОбороты` (нажать «+»)
2. Выделить таблицу → нажать кнопку **«>>»** (перенести все поля)
3. Нажать **«Ок»**

![[Pasted image 20260517143455.png]]

> [!note]
> **Отличие запросов 1С от SQL:**
> В 1С запросы используются **только для чтения данных**. Изменять (INSERT, UPDATE) данные через запросы нельзя. Это принципиальное отличие от SQL.

> Все они работают только с **числовыми значениями**. Дату в ресурс поместить нельзя.

## ⏱ Параметры периода в отчёте

### 📅 Способ 1: Два отдельных параметра с выражением «КонецПериода»

**Вкладка «Параметры»** в схеме КД содержит автоматически созданные параметры:
- **НачалоПериода**
- **КонецПериода**

**Проблема:** По умолчанию дата задаётся с нулевым временем (00:00:00). Записи, проведённые в течение дня (например, в 18:00), не попадают в выборку при указании этой даты как конечной.

**Решение — функция `КонецПериода`:**

В поле «Выражение» для параметра **«КонецПериода»** прописать:

```
КонецПериода(&КонецПериода, "День")
```

![[Pasted image 20260517151149.png]]

> [!info]
> **Разбор выражения:**
> - `КонецПериода` — функция языка СКД
> - `&КонецПериода` — значение, введённое пользователем в параметр (символ `&` = подстановка значения параметра)
> - `"День"` — строка, задающая, к концу какого периода приводить дату
>
> **Результат:** 12.09.2024 00:00:00 → 12.09.2024 **23:59:59**

**Варианты второго параметра:**

| Строка | Результат |
|---|---|
| `"День"` | Конец текущего дня (23:59:59) |
| `"Месяц"` | Конец текущего месяца |
| `"Год"` | 31.12.ГГГГ 23:59:59 |
| `"Час"` | Конец текущего часа (ЧЧ:59:59) |

**Настройка типа параметра (убрать время из отображения):**

Для обоих параметров: двойной клик → **«...»** → в «Состав даты» изменить с **«Дата и Время»** на **«Дата»** → Ок.

### 📅 Способ 2: Один параметр «СтандартныйПериод» (по заданию заказчика)

> [!tip]
> Более удобный вариант: одно поле выбора периода с встроенным календарём.

**Шаги:**

1. Стереть выражение `КонецПериода(...)` из параметра «КонецПериода»
2. Добавить новый параметр → Имя: **`Период`**
3. В типе нажать **«...»** → прокрутить вниз → выбрать **«СтандартныйПериод»** → Ок

**Настройка выражений для существующих параметров:**

В параметр **«НачалоПериода»** в поле «Выражение»:
```
&Период.ДатаНачала
```

В параметр **«КонецПериода»** в поле «Выражение»:
```
&Период.ДатаОкончания
```

> [!note]
> **Логика:** Пользователь вводит стандартный период → СКД берёт из него `ДатаНачала` и `ДатаОкончания` и подставляет в соответствующие параметры запроса.

**Скрытие исходных параметров от пользователя:**

На вкладке **«Параметры»** поставить галочки **«О»** (ограничение доступности) напротив:
- НачалоПериода
- КонецПериода

> [!info]
> Таким образом пользователь не сможет напрямую изменять эти параметры — управление происходит только через параметр «Период».

### 🔑 Ключевые понятия занятия

| Понятие | Описание |
|---|---|
| **Движения** | Перенос данных из документа в регистры при проведении |
| **Обработка проведения** | Обработчик события, в котором описывается алгоритм движений |
| **Конструктор движений** | Инструмент для автогенерации кода движений |
| **Записывать = Истина** | Свойство, разрешающее запись данных в регистр |
| **СКД** | Система компоновки данных — механизм построения отчётов |
| **Схема КД** | Инструкции: откуда брать данные и как их выводить |
| **Набор данных** | Источник данных в схеме КД (Запрос / Объект / Объединение) |
| **Конструктор запроса** | Визуальный инструмент для построения запросов (аналог SQL) |
| **Язык запросов 1С** | Декларативный язык, похожий на SQL, только для чтения данных |
| **Виртуальная таблица (Обороты)** | Агрегированное представление физической таблицы регистра |
| **Ресурсы в СКД** | Поля, по которым производится суммирование в группировках |
| **Рассчитать по** | Настройка уровней, на которых считается ресурс |
| **Макет оформления** | Предустановленная цветовая схема отчёта |
| **КонецПериода()** | Функция языка СКД для приведения даты к концу периода |
| **СтандартныйПериод** | Тип данных для задания диапазона дат одним полем |
| **&ИмяПараметра** | Синтаксис подстановки значения параметра в выражении СКД |
| **Быстрый доступ** | Режим отображения параметра прямо на форме отчёта |

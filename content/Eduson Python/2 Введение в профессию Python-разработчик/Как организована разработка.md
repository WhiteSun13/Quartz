# Какие есть методы управления проектами и командами
## Waterfall предполагает, что продукт проходит несколько последовательных стадий
Методология Waterfall предполагает такие **стадии развития продукта**:
- аналитика и дизайн;
- разработка; 
- тестирование; 
- публикация (выпуск); 
- поддержка. 

В Waterfall нельзя вернуться на предыдущие этапы.

## Waterfall используют, когда задачи и ресурсы заранее известны
**Waterfall применяют, когда заранее известны и зафиксированы:**
- сроки; 
- бюджет;
- результат.

Waterfall подходит для проектов в науке, госсекторе и небольших внутрикорпоративных проектов.

## Waterfall не позволяет гибко адаптироваться
Недостатки подхода:
- нельзя изменять закрытые этапы; 
- нельзя быстро адаптироваться  под изменения рынка.

## По Kanban сотрудники берут поступающие задачи

По методологии Kanban задачи берут   в работу с доски.

По мере выполнения задачу перемещают  по нескольким промежуточным этапам (выполнение, верификация, тестирование)   от «‎Сделать»‎ до «‎Сделано»‎.

Kanban подходит для технической поддержки уже существующих продуктов.
## По Scrum работу организуют повторяющимися спринтами

По методологии Scrum работа организована короткими регулярными циклами — спринтами, которые длятся от 1 недели   до 1 месяца (обычно 2 недели).

Команда ставит цель на спринт и задачи   для ее достижения. 

Спринты идут друг за другом.

## Во время спринта команда проводит церемонии

***Стендап (или дэйли):*** погрузить всех членов команды в контекст текущей работы   и помочь друг другу решить проблемы. 

***Груминг:*** сформулировать задачи для разработки на основе бизнес-запросов. 

***Планирование:*** оценить задачи и приоритизировать их. 

***Демо:*** показать коллегам результаты работы за спринт и собрать обратную связь. 

***Ретроспетива:*** обменяться впечатлениями  от спринта. 

По обратной связи команда может поменять приоритеты и переключиться на более важные задачи.
## Выбор методологии зависит   от конкретного продукта   и команды

**Waterfall** подходит для разработки внутрикорпоративных инструментов   и научных задач.

**Scrum** подходит для вывода на рынок  новых продуктов. 

**Kanban** подходит для проектов   по технической поддержке. 

Метод зависит от характеристик продукта  и опыта членов команды. Методология не влияет на содержание работы и инструменты, только на стиль управления и контроля на каждом этапе.

# Каков состав команды  разработки
## В команду разработки входят   разные специалисты

Основные роли в команде разработки:
- менеджер проектов;
- бэкенд-разработчик;
- фронтенд-разработчик;
- UI/UX-дизайнер;
- тестировщик
## Часть ролей выносят отдельно

Роли, которые выносят в отдельные подразделения:
- DevOps-инженер;
- Security-инженер;
- тестировщик.
Набор ролей зависит от проекта и размера компании-работодателя: в небольших компаниях ролей меньше.
# Какова роль бэкенд-  разработчика в команде
## Python-разработчик пишет  код для сервера и может брать дополнительные роли

Бэкенд-разработчик пишет внутреннюю логику работы сервера, который взиамодействует с другими бэкенд- сервисами и с фронтендом.
Дополнительные роли разработчика:
- Scrum-мастер;
- DevOps-инженер;
- Security-инженер;
- Фронтенд-разработчик.
## Главные soft-skills  для Python-разработчика

- слушать и слышать;
- задавать вопросы;
- формализовывать желания бизнеса в задачи разработки;
- быть на связи в рабочее время;помогать коллегам;
- договариваться и отстаивать интересы разработки.

Лучше, если в команде несколько Python- разработчиков. Разработчики могут учиться друг у друга, особенно если у них разная экспертиза. Также коллеги проверяют код друг друга.
# Из каких этапов состоит   процесс разработки
## Разработка состоит из 6 основных этапов

1. ***Формирование требований:*** команда понимает, что хочет сделать.
2. ***Проектирование:*** команда определяет, как будет выполнять задачи.
3. ***Реализация:*** команда пишет код, вычитывает его и проводит юнит-тестирование.
4. ***Тестирование:*** команда проверяет, что код работает без ошибок.
5. ***Внедрение:*** команда представляет результат конечным пользователям.
6. ***Эксплуатация и сопровождение:*** команда мониторит состояние продукта и отвечает на запросы службы поддержки.
